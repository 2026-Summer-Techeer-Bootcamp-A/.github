# COUNT(DISTINCT id)와 지도 API의 숨은 병목

**일정**: 2026-07-16  
**요약**: 배포 파이프라인과 관측 사각지대를 정리한 뒤 다시 k6로 300 VU 부하테스트를 돌리자, 목록과 피드, 기술 점유율 통계, 지도 엔드포인트가 전부 십수 초에서 수십 초까지 늘어졌고 에러율도 90퍼센트를 넘겼다. 원인을 좇는 과정에서 처음엔 인덱스 부재를 의심했지만, 실제로 EXPLAIN ANALYZE로 실행계획을 열어보니 진짜 범인은 훨씬 단순했다. 기본키를 대상으로 걸린 불필요한 DISTINCT 하나가 정렬 비용을 만들어내고 있었다.  

## 도입 배경

전날 배포 파이프라인의 겹겹이 쌓인 버그를 정리하고 우vicorn 워커 수를 9에서 4로 낮춘 뒤, 오염 없는 부하테스트 환경을 만들기 위해 타겟 서버와 분리된 소형 VM(`k6-loadgen`)을 새로 띄웠다. 이 VM에서 k6 stress 테스트를 300 VU까지 돌리자 서버가 사실상 무너졌다. 총 요청 9,241건 중 91.6퍼센트가 실패했고, 전체 p95는 43.6초까지 치솟았다. 흥미로운 점은 `healthz`처럼 DB를 거의 건드리지 않는 엔드포인트조차 87.6퍼센트가 실패했다는 것이었다. 이는 특정 쿼리 하나의 문제가 아니라 서버 전체가 밀려 있다는 신호였다.

## 1차 관측: 커넥션 풀 고갈과 DB CPU 100퍼센트

앱 컨테이너 로그를 열어보니 `sqlalchemy.exc.TimeoutError: QueuePool limit of size 30 overflow 10 reached, connection timed out` 에러가 반복되고 있었다. 워커 4개, 워커당 최대 40개 커넥션이니 전체 상한은 160개인데, 이 상한이 계속 바닥나고 있다는 뜻이었다. 그런데 앱 서버 CPU는 13퍼센트 남짓으로 여유로웠다. 컴퓨팅 부족이 아니라 순전히 커넥션 큐잉 문제라는 뜻이었다.

Cloud SQL의 실제 CPU 사용률을 Cloud Monitoring API로 직접 조회해보니, 평소 4퍼센트 수준이던 것이 부하 중 100퍼센트까지 치솟아 있었다. 이 DB는 `db-custom-2-7680`, 즉 2 vCPU짜리였다. 그런데 캐시 히트율은 99.99퍼센트였다. 디스크에서 읽어오는 게 아니라 전부 메모리 버퍼에서 처리되고 있다는 뜻이니, 디스크 I/O 문제가 아니라 순수한 연산 부하였다. 여기까지의 관측을 종합하면 결론은 하나로 좁혀졌다. 어떤 쿼리가 2개의 CPU 코어를 다 써버릴 만큼 무거운 연산을 하고 있고, 그 연산이 끝날 때까지 커넥션을 붙잡고 있어서 나머지 요청들이 큐에 밀려 쌓이고 있었다.

## 2차 관측: 기존 인덱스 확인

느린 엔드포인트 네 개, 즉 공고 목록(`postings_list`), 지도 핀(`postings_map`), 피드(`feed_postings`), 기술 점유율 통계(`stats_skill_share`)의 코드를 뜯어보니 처음엔 인덱스 부재가 원인으로 보였다. 이 네 엔드포인트가 공유하는 `posting` 테이블의 `is_deleted`, `close_date`, `region_district`, `industry`, `lat`, `lng` 컬럼에 인덱스가 없어 보였고, 반대로 캐시를 앞단에 둔 대조군(`/skills`, `/job_categories`)은 애초에 DB를 거의 안 건드려서 빨랐다.

그런데 실제 프로덕션 DB에 접속해 `pg_indexes`를 조회해보니 예상이 빗나갔다. `ix_posting_list_filter`, `ix_posting_coordinates`, `ix_posting_region_district` 같은 인덱스가 이미 다 만들어져 있었다. 더 나아가 EXPLAIN ANALYZE로 실제 쿼리를 단독 실행해보니, 공고 목록의 SELECT 쿼리는 2.4밀리초, 지도 핀 조회는 5.9밀리초 만에 끝났다. 인덱스는 문제가 아니었다. 이 판단을 뒤집는 데 결정적이었던 건 코드를 읽는 것만으로는 알 수 없었던 사실, 즉 실행계획을 직접 열어보는 일이었다.

## 3차 관측: COUNT(DISTINCT id)라는 함정

문제는 카운트 쿼리에 있었다. 공고 목록과 피드가 공유하는 페이지네이션용 총 건수 계산, 그리고 기술 점유율 통계에서 직군을 지정하지 않았을 때의 전체 표본 크기 계산이 전부 `SELECT COUNT(DISTINCT posting.id)` 형태로 짜여 있었다. `id`는 `posting` 테이블의 기본키이므로 애초에 유일한 값이다. 유일한 값에 DISTINCT를 거는 건 논리적으로 완전히 무의미하지만, 데이터베이스 입장에서는 다르다. postgres는 DISTINCT를 지키기 위해 매칭되는 행 전체를 정렬해서 중복을 제거해야 한다.

같은 쿼리를 EXPLAIN ANALYZE로 열어보니 이 사실이 숫자로 드러났다. 도메스틱 풀의 열려 있는 공고를 세는 쿼리는 총 638밀리초가 걸렸는데, 그중 605밀리초가 `Sort Method: external merge`, 즉 22만 건이 넘는 행을 디스크에 임시로 쓰면서까지 정렬하는 단계였다. 기술 점유율 통계 쪽은 더 심해서 920밀리초 중 885밀리초가 44만 건을 정렬하는 데 쓰이고 있었다. 인덱스를 잘 타고 있어도, 심지어 캐시에서 다 읽어와도, 결과 자체를 정렬하는 연산은 CPU를 그대로 쓴다. 이 사실이 이번 조사 전체를 관통하는 핵심이었다. 인덱스가 있다고 쿼리가 빠른 게 아니라, 그 인덱스로 찾아낸 결과를 가지고 무엇을 하느냐가 진짜 비용을 결정했다.

단독 실행에서는 638밀리초, 920밀리초 정도로 그쳤지만, 이게 문제로 보이지 않았던 이유는 동시성이 낮았기 때문이었다. 60에서 120 VU가 동시에 이 쿼리를 반복해서 두드리면, CPU 바운드인 정렬 작업 수십 개가 단 2개의 코어를 두고 경합하게 된다. 한 번에 638밀리초씩 걸리는 작업이 40개 동시에 몰리면 산술적으로 2개 코어로는 12초가 넘게 걸리고, 여기에 메모리 대역폭 경합과 컨텍스트 스위칭 비용까지 겹치면 실측치인 26초에서 51초까지 늘어나는 것도 이상하지 않았다. 부하가 낮을 때는 숨어 있다가 동시성이 오르면 증폭되어 드러나는 병목이었다.

## 4차 관측: 지도 API의 별도 비효율

`postings_map` 엔드포인트는 카운트 쿼리 문제와는 결이 다른 비효율을 갖고 있었다. 지도 화면 하나를 그리기 위해 핀 목록, 자치구별 클러스터, 히트맵을 각각 별도 쿼리로 조회하고 있었는데, 핀 조회에는 상한이 없어 좌표가 있는 공고 전체를 매번 애플리케이션으로 가져오고 있었다. 게다가 클러스터의 평균 좌표는 postgres의 GROUP BY와 AVG로 계산할 수 있는데도 개별 행을 전부 파이썬까지 끌고 와서 for 루프로 평균을 내고 있었다. 데이터가 늘어날수록 이 비효율은 선형으로 커질 구조였다.

## 해결

카운트 쿼리의 `COUNT(DISTINCT posting.id)`를 `COUNT(posting.id)`로 바꿨다. 두 쿼리 모두 posting 테이블 하나만 조회하고 JOIN으로 인한 행 증식이 없다는 걸 코드로 먼저 확인한 뒤였다. 이 한 줄짜리 변경만으로 불필요한 외부 정렬이 통째로 사라졌다.

지도 API는 세 가지를 함께 고쳤다. 핀 조회에 상한을 둬 데이터가 아무리 늘어도 요청 하나가 무한정 커지지 않게 했고, 클러스터의 평균 좌표 계산을 파이썬 반복문에서 postgres의 GROUP BY와 AVG로 옮겼으며, 기술 매칭 개수를 조회하는 IN절도 5천 건 단위로 청크를 나눠 대량 데이터에서도 안전하게 만들었다.

## 검증

같은 조건(120 VU, 램프업 15초, 플래토 45초)으로 수정 전후를 직접 비교했다. 전체 에러율은 5.68퍼센트에서 0.58퍼센트로 열 배 가까이 줄었다. 커넥션 풀 타임아웃으로 요청 전체가 실패하던 최악의 상황이 사실상 해소됐다는 뜻이다. `feed_postings`의 에러율은 42.86퍼센트에서 2.78퍼센트로, `postings_list`는 15.69퍼센트에서 1.54퍼센트로 떨어졌다.

다만 개별 쿼리의 평균 응답 시간 자체는 여전히 높았다. `postings_list`는 48.1초에서 42.0초로, `feed_postings`는 51.1초에서 44.4초로 소폭 개선되는 데 그쳤다. 왜 아직도 40초대가 남아 있는지는 곧바로 다음 조사로 이어졌다.

## 5차 관측: 옆에 남은 미수정 쿼리의 발목잡기

수정 배포 후 같은 조건으로 다시 확인해보니, 손도 대지 않은 `postings_nearby`가 17.2초, `stats_response_rate`가 14.0초, `stats_region_density`가 14.1초 평균을 보이고 있었다. 그런데 이 세 쿼리를 EXPLAIN ANALYZE로 단독 실행해보면 이야기가 또 달랐다. `postings_nearby`는 0.11밀리초, `stats_region_density`는 13.4밀리초, `get_hot_companies`는 6밀리초 만에 끝났다. 인덱스도 잘 타고 있었다. 딱 하나, `stats_response_rate`만 진짜 문제를 갖고 있었다.

이 엔드포인트는 도메스틱 풀의 응답률 데이터를 모아 분포를 보여주는데, 실제로 조회해보니 도메스틱 풀에는 `response_rate`가 채워진 행이 단 한 건도 없었다. 코드 주석에도 "wanted 소스만 response_rate를 적재하므로 표본이 얇다"고 이미 적혀 있었다. 표본이 0건이니 결과도 즉시 빈 값으로 나갈 텐데, 실행계획을 보니 그 0건을 확인하기 위해 매번 병렬 시퀀셜 스캔으로 테이블 전체(18만 8천 행 이상)를 다 훑고 있었다. `response_rate` 컬럼에 이 조건을 지원하는 인덱스가 없어서, "조건에 맞는 게 있는지 없는지"조차 테이블을 끝까지 읽어봐야 알 수 있었던 것이다. 게다가 이 스캔은 postgres의 병렬 워커 두 개를 동원해서 도는 병렬 시퀀셜 스캔이라, 실행되는 동안 DB의 2개 코어를 통째로 차지했다.

여기서 이번 조사 전체를 관통하는 두 번째 원칙이 드러났다. 한 요청이 아무리 빨리 끝나는 엔드포인트에 속해 있어도, 그 요청은 다른 모든 엔드포인트와 똑같은 커넥션 풀과 똑같은 2개의 CPU 코어를 나눠 쓴다. `postings_nearby`가 자기 몫의 일은 0.11밀리초 만에 끝내더라도, 바로 그 순간 `stats_response_rate`가 두 코어를 다 차지하고 있다면 `postings_nearby`의 요청은 커넥션을 얻기 위해, 혹은 CPU 차례를 얻기 위해 그만큼 대기해야 한다. 부하테스트가 28개 엔드포인트를 섞어서 동시에 두드리는 이상, 한 엔드포인트의 평균 응답 시간은 그 엔드포인트 자신의 효율만으로 결정되지 않는다. 이번에 고친 두 곳(카운트 쿼리, 지도 API)의 절대 지연이 크게 안 줄어든 이유는, 바로 옆에서 `stats_response_rate` 같은 다른 요청들이 여전히 코어를 붙잡고 있었기 때문이었다.

## 6차 관측: 여러 곳에 남은 DISTINCT 남용

`stats_response_rate`의 인덱스를 고치는 김에, 처음 발견한 COUNT(DISTINCT id) 패턴이 다른 곳에도 있는지 코드 전체를 훑었다. 결과는 절반씩 갈렸다. `posting.id`를 세는 두 군데(`_matched_skill_count`, `_required_skill_count`)와 `posting_tech.posting_id`를 세는 대여섯 군데(스킬 점유율, 개념-기술 상관, GitHub 스킬 매칭 등)에서 같은 낭비가 반복되고 있었는데, 전부 posting_tech나 posting_concept의 유니크 제약(같은 공고에 같은 스킬 또는 같은 개념이 두 번 걸릴 수 없다는 제약) 덕분에 DISTINCT가 애초에 아무 일도 하지 않는 자리였다. 이 자리들은 전부 DISTINCT를 걷어냈다.

다만 겉보기엔 비슷해 보여도 실제로는 DISTINCT가 꼭 필요한 자리도 있었다. `get_group_share`의 `union_count`가 그런 경우였는데, 이건 프레임워크 그룹(예: React, Vue, Angular)에 속한 여러 스킬을 한꺼번에 걸어놓고 그 스킬 중 하나라도 매칭되는 공고 수를 세는 쿼리였다. 한 공고가 React와 Vue를 동시에 요구한다면 posting_tech 조인에서 그 공고가 두 번 나타나므로, 이때는 DISTINCT를 빼면 실제로 공고 수를 부풀리는 버그가 생긴다. 기계적으로 전부 지우지 않고 각 쿼리가 GROUP BY로 묶는 단위와 유니크 제약이 실제로 중복을 막아주는지를 하나씩 확인한 다음에만 지웠다.

## 관측 대시보드에 남긴 자국

이번 조사에서 두 가지 지표가 특히 아쉬웠다. 하나는 앱이 자기 커넥션 풀을 얼마나 쓰고 있는지 보여주는 지표가 없었다는 점이다. Grafana에는 postgres 쪽 `numbackends`(DB가 보는 전체 커넥션 수)만 있었는데, 실제로 이번 사고의 방아쇠는 postgres의 400 커넥션 상한이 아니라 앱의 SQLAlchemy 풀 상한(워커당 40, 전체 160)이었다. 그래서 SQLAlchemy 엔진의 `pool.checkedout()` 값을 워커 전체에 대해 합산해서 노출하는 게이지(`db_pool_checked_out_connections`, `db_pool_capacity_connections`)를 새로 추가하고, Grafana에 앱 풀 사용률 패널과 시계열 패널을 붙였다. 이제 이 값이 postgres 쪽 커넥션 사용률보다 먼저 100퍼센트에 닿으면, 병목이 DB 자체가 아니라 앱의 풀 설정에 있다는 걸 바로 구분할 수 있다.

다른 하나는 k6의 VU 수를 서버 쪽에서 볼 방법이 없다는 점이다. VU는 순수하게 k6 클라이언트가 관리하는 개념이라 서버가 직접 알 방법이 없지만, Little's Law(동시 처리 중인 요청 수 = 처리율 × 평균 응답시간)로 근사할 수는 있다. 이미 있던 "동시 처리 중" 패널이 바로 이 계산이었는데, 그래프 하나로만 있어서 한눈에 읽기 어려웠다. 이 값을 단일 숫자로 보여주는 stat 패널을 새로 추가했다. 이 근사치는 k6 부하테스트 중에는 실제로 상당히 정확하다. 응답을 기다리며 막혀 있는 VU 수와 동시 처리 중인 요청 수가 거의 같기 때문이다. 반대로 실제 사람 트래픽에 이 패널을 쓸 때는 다음 절에서 설명할 보정이 필요하다.

## VU와 실제 사용자 수의 차이

부하테스트 결과를 보고 나면 자연스럽게 이런 질문이 따라온다. 300 VU가 서버를 무너뜨렸다면, 이건 실제 사용자 몇 명이 접속했을 때 무너진다는 뜻일까. 결론부터 말하면 VU 하나는 실제 사용자 한 명이 아니다.

`stress.js`의 각 VU는 요청 하나를 보내고 0.2초에서 0.7초만 쉬었다가 곧바로 다음 요청을 보낸다. 코드 주석에도 "load.js보다 짧게: VU 수로 부하를 만드는 게 목적"이라고 명시되어 있다. 반면 실제 사람은 페이지를 열고 읽고 스크롤하고 클릭하기까지 보통 십수 초에서 수십 초가 걸린다. 이 사이의 대기 시간을 흔히 think time이라고 부르는데, k6 VU의 think time은 사람보다 수십 배 짧다. 결국 VU 하나가 만들어내는 요청량은 실제 사용자 한 명이 아니라 동시에 브라우징 중인 사용자 수십 명 몫에 맞먹는다.

대략적인 환산은 이렇다. VU 하나는 반복 주기가 대기시간 평균(0.45초) 근처이므로 초당 약 2.2회 요청을 만든다. 실제 사용자 한 명은 페이지를 읽는 시간까지 포함하면 10초에서 20초에 한 번 정도 요청하는 것으로 잡는 게 합리적이다. 이 비율로 보면 VU 하나는 실제 사용자 스무 명에서 마흔 명 몫의 요청률에 해당하고, 120 VU는 대략 실제 동시 사용자 2천 명에서 4천 명이 쉴 새 없이 새로고침하는 수준의 부하가 된다. 부트캠프 데모나 포트폴리오 단계의 실제 트래픽 규모에서는 좀처럼 마주치기 어려운 수치다. 그렇다고 이번에 찾은 문제들이 가짜라는 뜻은 아니다. COUNT(DISTINCT id)나 인덱스 없는 시퀀셜 스캔은 트래픽 규모와 무관하게 존재하는 실제 비효율이었고, 다만 "지금 당장 실사용자 규모에서 터질 위험"이라기보다는 "트래픽이 늘었을 때를 대비한 여유를 미리 확보해둔 작업"으로 읽는 게 맞다. Grafana 화면에서 처리량이나 동시 처리 중 수치를 볼 때도, 그 뒤에 있는 VU 수가 실제 사용자 수와 그대로 대응하지 않는다는 걸 함께 감안해야 한다.

## 7차 관측: 하드웨어 문제 여부 확인

두 차례 수정을 거치고도 절대 지연이 크게 안 줄자, 자연스러운 다음 질문이 나왔다. 이건 결국 2 vCPU DB가 작아서 생기는 하드웨어 문제 아닌가. 답을 구하려면 이번 120 VU 테스트 도중 실제로 DB CPU가 얼마나 찍혔는지부터 봐야 했다.

Cloud Monitoring으로 확인해보니 이번 테스트 구간의 DB CPU는 최대 50.8퍼센트에 그쳤다. 처음 300 VU 테스트에서 100퍼센트까지 치솟았던 것과는 확연히 달랐다. app-vm의 CPU도 7에서 24퍼센트 선으로 여유로웠다. 두 서버 다 컴퓨트 자원이 바닥나지 않았는데 응답은 여전히 40초에서 60초씩 걸리고 있었다. 하드웨어가 문제라면 CPU가 100퍼센트에 붙어 있어야 하는데, 그렇지 않다는 게 오히려 이상했다.

그래서 테스트를 한 번 더 돌리면서 이번엔 postgres의 `pg_stat_activity`를 2초 간격으로 직접 들여다봤다. 락을 기다리는 세션 수(`wait_event_type = 'Lock'`)는 테스트 내내 0이었다. 락 경합은 아니라는 뜻이다. 반면 동시에 `active` 상태인 세션 수는 플래토 구간에서 116에서 121까지 치솟아 있었다. 2개의 코어를 가진 DB에 동시에 실행 중이라고 표시된 세션이 116개가 넘는다는 건, CPU 사용률 자체는 평균으로 보면 절반 수준이어도 실제로는 그 116개 세션이 2개 코어를 놓고 끊임없이 컨텍스트 스위칭하며 서로의 차례를 기다리고 있다는 뜻이다. Cloud Monitoring의 CPU 지표가 1분 단위 평균이라, 45초짜리 플래토 구간 동안 짧게 100퍼센트에 붙었다 떨어지기를 반복해도 그 평균값은 50퍼센트대로 뭉개져 보일 수 있다는 점도 함께 감안해야 한다.

결론적으로 이건 "하드웨어가 부족하다"와 "소프트웨어가 비효율적이다" 둘 중 하나로 깔끔하게 나뉘는 문제가 아니었다. 정확히는 세 번째 범주였다. 앱이 DB가 실제로 병렬 처리할 수 있는 양(코어 2개)보다 훨씬 많은 요청(동시에 100개 넘게)을 그대로 흘려보내고 있었고, DB는 그 많은 요청을 다 받아주려 애쓰다 코어를 나눠 쓰는 오버헤드에 시간을 쓰고 있었다. 코어를 늘리면 분명 도움이 되겠지만, 그전에 앱 단에서 동시에 DB로 내려보내는 요청 수 자체를 지금보다 적게 제한하고 나머지는 앱의 큐에서 기다리게 하는 것만으로도, DB가 실제로 처리할 수 있는 속도에 맞춰 흐름을 다듬을 수 있다. 어느 쪽이 더 효과적인지는 이번 조사의 범위를 넘어서는 다음 과제로 남겨둔다.

이 과정에서 계측 자체의 버그도 하나 찾았다. 앞서 추가한 `db_pool_checked_out_connections` 게이지가 테스트 내내 0만 보여주고 있었는데, 이는 실제로 커넥션이 안 쓰이고 있어서가 아니라 게이지를 `Gauge.set_function(lambda: engine.pool.checkedout())` 방식으로 만든 탓이었다. `set_function`은 그 Gauge 객체의 `collect()`가 해당 프로세스 안에서 직접 호출되어야 값을 계산하는데, `/metrics` 핸들러는 워커마다 이미 파일에 써놓은 값을 `MultiProcessCollector`로 읽어올 뿐 로컬 Gauge의 `collect()`를 부르지 않는다. 즉 이 값은 파일에 기록될 기회 자체가 없었다. SQLAlchemy 풀의 checkout과 checkin 이벤트에 걸어 `inc()`와 `dec()`로 바꾸니, 호출 즉시 각 워커의 공유 파일에 기록되어 정상적으로 합산됐다. 지표를 만들어놓고 검증 없이 믿었다면 다음번에도 "커넥션 풀은 여유롭다"는 잘못된 결론을 낼 뻔한 대목이었다.

## 8차 관측: 하드웨어도 스로틀링도 아닌 원인

코어 수보다 훨씬 많은 요청이 동시에 몰린다는 그림까지는 나왔는데, 그다음 선택지를 두고 다시 생각해봤다. DB 코어를 늘리는 것과, 앱이 DB로 흘려보내는 동시 요청 수 자체를 줄이는 것 사이에서 고르는 대신, 애초에 왜 이렇게 많은 요청이 매번 DB까지 가야 했는지를 물었다.

답은 간단했다. `stats_region_density`, `stats_hot_companies`, `stats_response_rate`, `postings_map`이 반환하는 값은 posting 데이터가 스크래핑 주기로만 바뀌는 통계인데, 요청이 올 때마다 매번 DB에서 다시 계산하고 있었다. 이미 코드 안에 정답이 있었다. `/skills`, `/job_categories`가 부하테스트 내내 계속 빨랐던 건 정확히 이 이유였는데, `reference_cache.py`라는 Redis 캐시가 앞을 막고 있어 대부분의 요청이 DB를 아예 안 건드렸기 때문이다. 같은 패턴을 이 네 엔드포인트에도 그대로 적용했다. `postings_map`만은 조심할 지점이 있었는데, 이력서 기반 매칭률(`resume_id`/`session_id`)이 섞이면 사용자마다 응답이 달라지므로, 그런 개인화 파라미터가 없는 익명 조회 경로만 캐시하도록 분기했다.

캐싱은 애초에 DB까지 가는 요청 자체를 줄이는 조치이고, 그래도 DB까지 가야 하는 나머지 요청을 위해 PgBouncer를 트랜잭션 풀링 모드로 앱과 DB 사이에 뒀다. 앱은 지금처럼 워커당 40개, 전체 160개의 커넥션을 그대로 유지해도, 실제 Postgres 백엔드로 나가는 연결 수는 PgBouncer가 훨씬 적게(`DEFAULT_POOL_SIZE=20`) 유지해준다. 앞서 관측한 "116개 세션이 코어 2개를 두고 컨텍스트 스위칭하는" 증상을, 앱의 동시성을 줄이지도 DB 코어를 늘리지도 않고 그대로 없앨 수 있는 조치다.

두 조치의 역할은 다르다. 캐싱은 애초에 DB에 도달하는 요청의 절대량을 줄이고, PgBouncer는 그래도 DB에 도달하는 요청들이 실제 Postgres 백엔드를 두고 벌이는 경합을 줄인다. 하드웨어를 늘리는 것보다 두 조치 모두 비용이 들지 않고, "안 바뀌는 데이터를 매번 다시 계산한다"는 진짜 원인에 더 가깝게 맞닿아 있다.

## 결론

이번 조사는 하나를 고치고 끝나는 이야기가 아니었다. 처음 찾은 COUNT(DISTINCT id)는 확실한 원인이었지만 전체 그림의 일부였고, 그 자리를 고친 뒤에도 남아 있던 지연은 아직 손대지 않은 다른 엔드포인트가 같은 DB와 같은 커넥션 풀을 붙잡고 있었기 때문이었다. 인덱스가 없어서 매번 테이블 전체를 훑고 있던 `stats_response_rate`를 찾아 고치고, 같은 패턴의 DISTINCT 남용을 코드 전체에서 골라내면서도 정말 필요한 자리(`get_group_share`의 union_count)는 남겨두는 판단까지 거친 뒤에야, 병목의 대부분을 실제로 설명할 수 있었다. 그 뒤에도 남은 지연을 두고 하드웨어 탓인지 물었을 때, 답은 CPU 사용률과 동시 활성 세션 수를 함께 봐야만 나왔다. DB 코어가 놀고 있는 것도 아니고 완전히 포화된 것도 아닌, "코어 수보다 훨씬 많은 요청을 동시에 흘려보내고 있다"는 세 번째 그림이었다. 마지막으로 이 그림을 두고 하드웨어를 늘릴지 앱의 동시성을 줄일지 고르는 대신, 애초에 안 바뀌는 통계를 매번 다시 계산하고 있었다는 더 근본적인 원인으로 돌아가 Redis 캐싱을 붙이고, 그래도 DB까지 가는 나머지 요청을 위해 PgBouncer로 실제 백엔드 커넥션 수를 조절했다. 관측 대시보드에는 앱 쪽 커넥션 풀 사용률과 VU 추정치를 추가해, 다음에 비슷한 상황이 오면 DB 커넥션 수만 보고 헛다리를 짚지 않도록 해두었다.

## 방법론 메모

이번 조사에서 되풀이해서 확인한 원칙은, 코드를 읽고 세운 가설을 실제 실행계획으로 검증하지 않으면 틀린 결론에 도달할 수 있다는 것이었다. 인덱스가 없어 보인다는 코드 리뷰만으로는 실제로 인덱스가 이미 존재하고 잘 활용되고 있다는 사실을 알 수 없었다. `EXPLAIN (ANALYZE, BUFFERS)`로 실제 프로덕션 DB에 대고 쿼리를 직접 돌려본 다음에야, 문제가 인덱스가 아니라 결과를 정렬하는 방식에 있다는 걸 확인할 수 있었다. 부하테스트로 증상을 재현하고, 관측 지표(커넥션 수, 캐시 히트율, CPU 사용률)로 병목의 위치를 좁힌 다음, 실행계획으로 정확한 원인을 확정하고, 수정 후 같은 조건으로 재현해 개선 폭을 수치로 확인하는 순서를 지킨 것이 이번 조사가 헛다리를 짚지 않은 이유였다.

### 각주: 이 문서에 쓰인 용어

**DISTINCT**란 SQL에서 조회 결과 중 중복된 값을 제거하고 유일한 값만 남기라는 지시어다. 예를 들어 한 테이블에 같은 이름이 여러 번 나온다면 `SELECT DISTINCT name`은 그 이름을 한 번씩만 보여준다. 문제는 이 중복 제거를 하려면 데이터베이스가 내부적으로 값을 비교할 수 있도록 정렬하거나 해시 테이블을 만들어야 한다는 점이다. 만약 애초에 중복될 수 없는 값, 즉 기본키처럼 이미 유일함이 보장된 값에 DISTINCT를 걸면, 정렬 비용만 그대로 발생하고 실제로 걸러지는 중복은 하나도 없다.

**기본키(primary key)**란 테이블의 각 행을 유일하게 식별하는 값이다. 이 값은 정의상 절대 중복되지 않는다. 이번 사례의 `posting.id`가 여기 해당한다.

**EXPLAIN ANALYZE**는 postgres에게 어떤 쿼리를 실제로 실행하면서 그 안에서 어떤 단계를 거쳤는지, 각 단계가 몇 밀리초 걸렸는지를 보여달라고 요청하는 명령이다. 쿼리가 인덱스를 탔는지, 테이블 전체를 훑었는지, 정렬이나 집계에 얼마나 시간을 썼는지를 코드만 봐서는 알 수 없고 이 명령으로 직접 확인해야 한다.

**인덱스(index)**는 책의 찾아보기처럼, 테이블 전체를 처음부터 훑지 않고도 원하는 행을 빠르게 찾을 수 있도록 미리 만들어둔 자료구조다. 인덱스가 있으면 특정 조건에 맞는 행을 찾는 속도는 빨라지지만, 찾아낸 행들을 가지고 정렬이나 집계 같은 추가 연산을 하는 비용까지 없애주지는 않는다.

**시퀀셜 스캔(sequential scan)**은 인덱스를 쓰지 않고 테이블의 모든 행을 처음부터 끝까지 순서대로 훑는 방식이다. 조건에 맞는 행이 테이블의 상당 부분을 차지할 만큼 많다면, postgres는 인덱스를 쓰는 것보다 오히려 순차적으로 다 훑는 편이 더 빠르다고 판단해 일부러 시퀀셜 스캔을 선택하기도 한다.

**캐시 히트율(cache hit ratio)**은 데이터를 읽을 때 디스크까지 가지 않고 메모리(버퍼 캐시)에서 바로 찾아낸 비율이다. 이 값이 100퍼센트에 가깝다는 건 디스크 입출력이 느려서가 아니라, CPU가 순수 연산에 시간을 쓰고 있다는 신호다.

**유니크 제약(unique constraint)**은 테이블의 특정 컬럼 조합이 절대 중복될 수 없도록 데이터베이스가 강제하는 규칙이다. 이번 사례의 posting_tech 테이블은 (posting_id, skill_id) 조합에 유니크 제약이 걸려 있어, 같은 공고에 같은 기술이 두 번 연결되는 일이 애초에 불가능하다. 이 보장이 있으면, 특정 공고 하나 또는 특정 기술 하나로 좁혀놓은 조회에서는 DISTINCT 없이도 중복이 생기지 않는다는 걸 코드가 아니라 스키마 제약으로 증명할 수 있다.

**VU(Virtual User)**는 k6 같은 부하테스트 도구가 만들어내는 가상의 동시 접속자다. VU 하나는 정해진 시나리오(요청을 보내고 얼마간 쉬었다가 다시 보내는)를 반복하는 독립된 실행 흐름이며, 실제 사람 한 명이 아니라 부하를 만들어내는 단위에 가깝다. VU 수와 실제 동시 사용자 수의 관계는 시나리오에 설정된 대기 시간에 따라 크게 달라진다.

**think time**은 실제 사용자가 화면을 읽거나 다음 행동을 결정하기까지 걸리는, 요청과 요청 사이의 대기 시간이다. 부하테스트 시나리오가 이 시간을 실제 사용자보다 훨씬 짧게 잡으면, 같은 VU 수라도 훨씬 많은 실제 사용자를 흉내내는 것과 같은 효과를 낸다.

**Little's Law**는 대기 행렬 이론의 기본 공식으로, 시스템 안에 평균적으로 몇 개의 요청이 동시에 머물러 있는지를 처리율과 평균 체류 시간의 곱으로 계산한다(동시 처리 중인 요청 수 = 처리율 × 평균 응답시간). 각 요청이 무엇을 하는지 직접 세지 않고도, 이미 측정하고 있는 두 지표만으로 동시성을 추정할 수 있다는 게 이 공식의 쓸모다.

**컨텍스트 스위칭(context switching)**은 CPU 코어 하나가 여러 프로세스를 번갈아 실행하기 위해 지금 처리하던 작업의 상태를 저장하고 다음 작업의 상태를 불러오는 과정이다. 코어 수보다 훨씬 많은 프로세스가 동시에 실행을 기다리면, 각 프로세스가 실제로 계산하는 시간보다 서로 자리를 바꾸는 데 드는 시간이 커져서 전체 처리량이 떨어진다. CPU 사용률 수치만 봐서는 이 오버헤드가 잘 드러나지 않는다는 게 까다로운 점이다.

**pg_stat_activity**는 postgres가 현재 맺고 있는 모든 연결의 상태를 보여주는 시스템 뷰다. 각 연결이 지금 실제로 쿼리를 실행하는 중(active)인지, 아무 일도 안 하고 대기 중(idle)인지, 트랜잭션을 열어놓은 채 쉬고 있는지(idle in transaction), 무엇을 기다리고 있는지(wait_event, wait_event_type)까지 실시간으로 확인할 수 있다.
