# 성능과 CI/CD 최적화 통합 요약

프론트엔드, 백엔드 CI/CD, 백엔드 부하테스트 세 갈래로 진행된 최적화 작업을 문제, 원인, 해결법 위주로 압축했다. 각 항목의 전체 서사와 실측 로그는 해당 저널 원문(`frontend-perf-journal/`, `backend-cicd/`, `perf-build-journal/`)에 있다.

## 프론트엔드

**번들 크기와 초기 로딩(00~02번)**: Lighthouse 성능 점수가 50점이었고 FCP 8.7초, LCP 9.6초로 첫 화면이 뜨는 데만 한참 걸렸다. 원인은 d3, echarts, leaflet, highlight.js 같은 무거운 라이브러리가 전부 하나의 index.js(3.98MB)에 뭉쳐 있었고, 90개 넘는 라우트가 전부 정적 import라 앱을 켜는 순간 전체를 다 받아야 했기 때문이었다. vite의 manualChunks로 라이브러리를 분리하고 라우트를 lazy import로 바꿔, index.js를 1.18MB(70퍼센트 감소)로 줄였다.

**라우트 전환 시 흰 화면 깜빡임(03번)**: 탭을 옮길 때마다 화면 전체가 흰 화면으로 깜빡였다. 최상위에 Suspense를 하나만 둬서 셸 레이아웃(헤더, 네비게이션)까지 함께 로딩 화면으로 가려졌던 게 원인이었다. Suspense를 라우트 콘텐츠 영역(Outlet) 안으로 옮기고 서브페이지들을 개별 dynamic import로 바꾸자, 셸은 그대로 유지된 채 콘텐츠만 전환되도록 고쳐졌다. 최종적으로 index.js는 30.33KB(초기 대비 97퍼센트 감소)까지 줄었고 Lighthouse 점수는 61점, FCP 6.3초, LCP 6.7초로 개선됐다.

## 백엔드 CI/CD

**배포 게이트 부재(00~01번)**: `main` 브랜치에 병합되면 테스트가 실행되지도 않은 채 그대로 배포되고 있었다. `ci.yml`은 main을 제외하고 도는 워크플로였고 `deploy.yml`은 main 전용으로 따로 돌아, 둘 사이에 아무 연결고리가 없었기 때문이다. SQLite 기반 빠른 테스트와 실제 Postgres(pgvector, CITEXT 포함) 기반 통합 테스트를 나누고, `deploy` 잡에 `needs: test`를 걸어 테스트를 통과해야만 배포가 이어지도록 게이트를 도입했다.

**파이프라인 속도(02~04번)**: PR CI가 6분 9초, 배포가 15분 58초씩 걸렸다. 매 잡마다 의존성을 처음부터 다시 설치하고, 모든 단계가 완전히 직렬로 돌고, 도커 레이어 캐시도 안 쓰고 있었던 탓이다. ruff 전용 액션과 venv 캐싱, Buildx의 GHA 캐시, 독립적인 잡들의 병렬화를 적용해 배포 시간을 2분 50초까지 줄였다(82.3퍼센트 감소, 실배포로 검증 완료).

**이미지 크기와 빌드 자원(03번)**: 이미지가 2.94GB였고, 유닛 테스트를 돌릴 때마다 `sentence-transformers` 같은 무거운 ML 의존성까지 매번 재설치되고 있었다. 단일 스테이지 Dockerfile이라 테스트용 의존성과 런타임 의존성이 분리되지 않았던 게 원인이다. `requirements-test.txt`를 따로 빼고 멀티스테이지 빌드로 전환해 이미지를 212MB(92.7퍼센트 감소)까지 줄였고, 모델 캐시를 볼륨으로 마운트해 중복 다운로드도 없앴다.

## 백엔드 부하테스트와 성능

**공고 목록 조회 크래시(02~03번)**: 프로덕션에 VU 3짜리 가벼운 부하만 걸어도 오류율 11퍼센트, p95 24.94초가 나왔다. 원인은 필터 없이 공고 55만 건 전체를 하나의 SQL IN절에 그대로 바인딩해, PostgreSQL의 파라미터 개수 한도(65,535)를 넘겨버렸기 때문이었다. 매칭이 필요 없는 경로는 DB 레벨 LIMIT/OFFSET으로, 매칭이 필요한 경로는 5000건 단위 청크 처리로 바꿔 39.78초짜리 요청을 0.31초로 줄였다.

**커넥션 풀 경합과 지연 전이(04~06번)**: `stats` 계열 무거운 집계 엔드포인트 5종의 p95가 4~12초였는데, 이 무거운 쿼리들이 커넥션 풀 슬롯을 오래 붙잡고 있어서 전혀 무관한 가벼운 API 요청까지 도미노처럼 지연되는 현상이 관찰됐다. 원인은 materialized view 없이 매 요청마다 raw aggregation을 다시 계산하고 있었기 때문으로, MV 신설과 SQL 재작성, Redis 캐싱을 조합한 계획을 세웠다.

**관측 자체의 사각지대(07, 10번)**: 워커를 늘리고 VU를 200까지 올려도 처리량이 초당 2.36건에서 멈췄는데, 원인 중 하나는 uvicorn에 `--workers` 옵션 자체가 빠져 워커가 하나뿐이었다는 것이었다. 더 심각했던 건 Grafana의 p95 그래프가 항상 1000ms에 고정되어 있던 문제였는데, `prometheus_fastapi_instrumentator`의 히스토그램 버킷 상한이 1초뿐이라 그 이상의 모든 지연이 다 뭉개져 보이지 않았던 것이었다. 버킷을 60초까지 넓히자 그제야 실제로 8개 엔드포인트가 6초에서 54초까지 심각하게 느리다는 사실이 드러났다.

**raw aggregation 8종 최적화(11번)**: 10번에서 드러난 8개 엔드포인트는 MV 없이 조인 집계를 매번 새로 계산하거나, ILIKE 부분 검색이 인덱스 없이 풀스캔을 돌거나, 좌표 범위 조회에 인덱스가 없는 게 원인이었다. materialized view 신설, `pg_trgm` GIN 인덱스, 복합 B-Tree 인덱스를 추가해 전 엔드포인트 p95를 1.3초 미만으로 낮췄다(`postings_search`는 670배 개선).

**조용히 실패하던 배포(12번)**: DB CPU가 90퍼센트인데 처리량은 초당 1~2건에 그쳤고, 그런데도 GitHub Actions는 계속 success로 표시되고 있었다. 배포 스크립트가 `tee`로 파이프되어 있어 실제 배포 실패(디스크 부족)가 종료 코드에 반영되지 않았고, 여기에 buildx 컨테이너의 gcloud 인증 문제, PyTorch CUDA wheel 오설치, 멀티워커 환경에서 `/metrics`가 워커 하나 값만 보여주는 문제, 로컬/VM에서 돌린 부하테스트 자체가 네트워크와 자원 경합으로 오염된 문제까지 네 겹으로 쌓여 있었다. `set -o pipefail`, 이미지 prune, CPU 전용 wheel 인덱스, `PROMETHEUS_MULTIPROC_DIR` 기반 멀티프로세스 집계, 타겟과 분리된 전용 부하생성 VM으로 하나씩 해소했다.

**COUNT(DISTINCT id)와 남은 병목(13번)**: 인덱스는 이미 다 있었는데도 목록, 피드, 통계 엔드포인트가 여전히 느렸던 원인은 이미 유일한 값인 기본키에 불필요하게 걸린 `COUNT(DISTINCT id)`가 매번 수십만 건을 정렬하고 있었기 때문이었다. 이 정렬을 없애고 같은 패턴의 DISTINCT 남용을 코드 전체에서 걷어낸 뒤, 인덱스가 없어 매번 테이블 전체를 훑던 `stats_response_rate`도 고쳤다. 그 뒤에도 남은 지연을 놓고 하드웨어 문제인지 확인해보니, DB 코어는 반쯤 남아 있는데 동시 활성 세션이 116개를 넘어 코어 2개를 두고 컨텍스트 스위칭에 시달리고 있었다. 근본적으로는 안 변하는 통계를 매번 다시 계산하고 있었던 것이 문제라, `stats_region_density`, `stats_hot_companies`, `stats_response_rate`, `postings_map`에 Redis 캐싱을 붙이고, 앱과 DB 사이에 PgBouncer를 트랜잭션 풀링 모드로 두어 실제 Postgres 백엔드 연결 수 자체를 줄였다.

## 전체적으로 반복된 교훈

거의 모든 사례에서 공통된 패턴이 하나 있었다. 겉보기 증상(느림, CPU 사용률, 에러율)만으로 원인을 짐작하면 자주 틀렸고, 실행계획(EXPLAIN ANALYZE)이나 관측 지표(pg_stat_activity, 실제 배포 이미지 확인)를 직접 열어봐야 진짜 원인이 나왔다. 그리고 관측 도구 자체가 거짓말을 하는 경우(히스토그램 버킷 상한, tee 마스킹, 멀티프로세스 미집계)가 여러 번 있었다는 점도 눈에 띈다. 무언가를 고쳤다고 결론 내리기 전에 그 결론을 만들어낸 측정 도구부터 의심하는 습관이, 이번 최적화 작업 전체를 관통하는 가장 중요한 태도였다.
