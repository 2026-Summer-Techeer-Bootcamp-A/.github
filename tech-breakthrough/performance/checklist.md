# 성능 개선 체크리스트 (MV / Redis) — 팀원 A/B/C/D 분담

**일정**: 2026-07-14  
**배경**: `performance-test/k6/` 5종 부하테스트(스모크/일반부하/과부하/중단점/스파이크)로 안전한 GET 엔드포인트 31개를 실측하는 과정에서, `/api/v1/stats/*` 계열 5개 엔드포인트가 materialized view 없이 `posting`/`posting_tech`를 직접 조인해 매 요청마다 GROUP BY를 돌리고 있고, 이게 몇 초씩 DB 커넥션을 붙잡아 나머지 가벼운 엔드포인트까지 도미노로 느려지는 걸 확인했다. 자세한 실측 수치와 원인 추정은 [`perf-build-journal/04-endpoint-coverage-expansion-and-connection-pool-finding.md`](perf-build-journal/04-endpoint-coverage-expansion-and-connection-pool-finding.md), 오늘 로컬+프로덕션 전체 실행 결과는 [`perf-build-journal/05-2026-07-14-local-and-production-results.md`](perf-build-journal/05-2026-07-14-local-and-production-results.md) 참고.

이 문서는 그 개선 작업(MV 신설, Redis 캐싱)을 4명이 **같은 파일을 동시에 건드리지 않도록** 나눈 작업 배정표다. 구현 방법은 담당자 재량이고, 아래 체크리스트는 "무엇이 되어 있어야 하는지"만 확인한다.

## 용어

- **MV (Materialized View, 구체화 뷰)**: 쿼리 결과를 매번 다시 계산하는 대신 미리 계산해서 저장해두는 테이블 형태의 객체. 무거운 조인·집계를 한 번만 돌려서 그 결과를 저장해두면, 이후 조회는 그 저장된 결과를 읽기만 하면 되어 훨씬 빠르다. 단점은 원본 데이터가 바뀌어도 자동으로 따라 바뀌지 않는다는 것 — `REFRESH`라는 명령을 실행해줘야 최신 상태가 된다.
- **Redis**: 디스크가 아니라 메모리에 데이터를 저장하는 key-value 저장소. Postgres 같은 일반 DB보다 훨씬 빠르게 읽고 쓸 수 있어서, "계산 비용은 큰데 자주 조회되는 결과"를 임시로 저장해두는 캐시 용도로 흔히 쓴다. 이 프로젝트에서는 이미 로그아웃 토큰 관리, 이력서 확인 세션, 뉴스 피드 캐싱에 쓰이고 있다.
- **TTL (Time To Live)**: 캐시에 저장한 값이 자동으로 사라지기까지의 유효 시간. Redis에 값을 넣을 때 TTL을 같이 지정해두면, 그 시간이 지난 뒤에는 값이 자동으로 삭제(만료)되고 다음 요청부터는 원본을 다시 조회해서 새로 캐싱한다. TTL을 짧게 잡으면 최신 데이터를 더 자주 반영하지만 캐시 효과(DB 부담 감소)는 줄어들고, 길게 잡으면 그 반대다.

## 먼저 참고할 것: 기존 패턴

- **MV**: `mv_skill_share`, `mv_cooccurrence`가 이미 있다. 앱 기동 시 자동 생성되고, 수집기(collector) 작업이 끝날 때마다 자동으로 REFRESH된다. 이 프로젝트에는 alembic 같은 마이그레이션 도구가 없어서, MV를 새로 만드는 것 자체가 "생성 + 갱신 훅 등록"으로 끝난다.
- **Redis**: `news.py`가 이미 이 패턴이다(4시간 TTL로 외부 API 응답을 캐싱). 공용 Redis 클라이언트가 이미 있으니 그걸 재사용한다.

## 공통 규칙 (충돌 방지)

- **자기 담당 엔드포인트/함수만 수정한다.** 여러 명이 같은 파일(예: MV 등록·갱신 관련 공용 파일)에 손대야 하는 경우, 자기 것만 끝에 추가하고 남의 코드는 건드리지 않는다.
- **PR은 작게, 자주 올린다.** 담당한 엔드포인트 하나 끝날 때마다 바로 PR을 올린다.
- **머지 순서가 걱정되면** 같은 공용 파일을 만지는 A, B가 먼저 순서를 조율한다. C, D는 그 공용 파일을 아예 안 건드리므로 A/B와 충돌할 일이 없다.
- **검증은 k6로 한다.** 로컬 dev 스택에서 스모크 → 일반부하 순서로 돌려서, 담당 엔드포인트의 `p95`가 눈에 띄게 줄었는지 확인한다.
- **전부 병합된 뒤** `load.js`/`stress.js`의 threshold(지금은 느슨하게 잡혀 있다)를 다시 좁히는 후속 작업이 있다 — 이 체크리스트 범위 밖.

## 팀원 A — MV 신설: 국내/해외 격차, 연도별 기술 트렌드

**목표**: `/api/v1/stats/global-domestic-gap`, `/api/v1/stats/skill-trend-yearly`

**왜 이 둘인가 (실측 p95)**

| 엔드포인트 | 로컬(VU 20/1분) | 프로덕션(VU 5/20s) |
|---|---|---|
| `stats_global_domestic_gap` | 4909ms | 1555ms |
| `stats_skill_trend_yearly` | 12353ms | 5652ms |

둘 다 매 요청마다 posting 전체 코퍼스를 다시 훑어 집계하는 구조라, `mv_skill_share`와 같은 이유로 MV 대상이다.

**만지는 영역**: 이 두 엔드포인트의 조회 로직(CRUD 레이어) + MV 생성/갱신 등록(공용 파일, B와 겹치는 지점 — 자기 블록만 추가)

**체크리스트**
- [ ] 두 엔드포인트 각각 응답을 만드는 무거운 집계가 MV로 사전 계산되게 했는가?
- [ ] 앱을 처음 띄울 때 두 MV가 자동으로 만들어지는가(수동 마이그레이션 없이)?
- [ ] 수집기 작업이 끝날 때마다 두 MV가 자동으로 최신화(REFRESH)되는가?
- [ ] 두 엔드포인트가 원본 테이블이 아니라 새 MV를 조회하도록 바뀌었는가?
- [ ] 응답 스키마(필드 이름·타입)가 이전과 완전히 동일한가?
- [ ] 기존 자동화 테스트가 전부 통과하는가?
- [ ] 로직 변경 전후로 같은 조건에 대한 응답 값이 동일한가(회귀 없음)?
- [ ] k6 스모크/일반부하로 두 엔드포인트의 p95가 확실히 줄어든 걸 확인했는가?

**갱신 정책**: TTL이 아니라 "REFRESH 시점" 개념 — 수집기 작업 성공 시 자동 갱신된다.

## 팀원 C — 쿼리 재작성 + Redis 캐싱: 채용 성수기 지수

**목표**: `/api/v1/stats/hiring-season`

**왜 이건 MV가 아니라 쿼리 재작성인가**: A/B와 성격이 다르다. 무거운 조인이 아니라, DB에서 처리해야 할 집계(월별 카운트)를 애플리케이션이 전체 행을 다 가져와 파이썬으로 계산하는 구조다(03번 build-journal에서 고친 페이지네이션 버그와 같은 계열의 안티패턴). MV까지 갈 필요 없이 집계를 DB 쿼리 안으로 옮기는 게 먼저다. 실측 p95는 로컬 2599~2929ms.

**만지는 영역**: 이 엔드포인트의 조회 로직(CRUD 레이어) + 이 엔드포인트의 라우터 핸들러(A/B/D와 겹치지 않음)

**체크리스트**
- [ ] 월별 집계가 애플리케이션이 아니라 DB 쿼리(GROUP BY) 안에서 이루어지도록 바뀌었는가?
- [ ] 재작성 전후로 응답 값이 완전히 동일한가(회귀 없음)?
- [ ] 응답이 Redis 캐시에 적재되게 했는가?
- [ ] TTL은 6시간으로 설정했는가?
- [ ] 캐시 히트/미스 양쪽 다 올바른 응답 스키마로 나가는지 확인했는가?
- [ ] 기존 자동화 테스트가 전부 통과하는가?
- [ ] 같은 요청을 연달아 두 번 보냈을 때 두 번째가 캐시로 훨씬 빨라지는 걸 확인했는가?
- [ ] k6 스모크/일반부하로 p95 개선을 확인했는가?

**TTL**: 6시간. `news.py`의 4시간보다 조금 길게 잡았다 — 이 데이터는 수집기 실행 주기(대략 하루)보다 훨씬 자주 바뀌지 않는다. 캐시는 수집기 REFRESH와 자동 연동되지 않으므로(TTL 기반 만료), 새 수집 직후 최대 6시간은 이전 데이터가 보일 수 있다는 점은 감안한다.

## 팀원 D — Redis 캐싱: 자주 호출되는 참조성 데이터

**목표**: `/skills`, `/api/v1/job-categories`, `/api/v1/certs`, `/api/v1/company/by-skill`

**왜 이 넷인가**: 개별 쿼리 자체는 A/B/C의 대상만큼 무겁지 않지만(로컬 p95가 대략 skills 13~1877ms, job-categories 64~1700ms, certs 10~473ms, company-by-skill 23~1744ms 범위), 동시 요청이 몰리면 04번 문서에서 확인한 DB 커넥션 경합 때문에 이런 가벼운 쿼리까지 같이 느려진다. 넷 다 프론트에서 자주 불리는데 수집기가 돌 때만 바뀌는 참조성 데이터라 캐싱 효율이 좋다. 이 엔드포인트 자체가 빨라지는 것도 있지만, **DB 왕복을 줄여 A/B/C가 고치는 무거운 쿼리들의 경합 상대를 줄이는 효과**가 더 크다.

**만지는 영역**: 이 네 엔드포인트의 라우터 핸들러(A/B/C 중 누구와도 겹치지 않음)

**체크리스트**
- [ ] 네 엔드포인트 각각 응답이 Redis 캐시에 적재되게 했는가?
- [ ] 캐시 키가 파라미터별로 구분되는가(같은 엔드포인트라도 파라미터가 다르면 다른 캐시로 취급)?
- [ ] `skills`/`job-categories`/`certs`는 TTL 24시간으로 설정했는가?
- [ ] `company-by-skill`은 TTL 6시간으로 설정했는가?
- [ ] 캐시 히트/미스 양쪽 다 올바른 응답 스키마로 나가는지 확인했는가?
- [ ] 기존 자동화 테스트가 전부 통과하는가?
- [ ] 같은 요청을 연달아 두 번 보냈을 때 캐시 히트로 빨라지는 걸 확인했는가?
- [ ] k6 스모크/일반부하로 이 네 엔드포인트뿐 아니라 **다른 엔드포인트들의 p95도 같이 좋아졌는지**(경합 완화 효과) 확인했는가? — 이게 D 작업의 진짜 목표다.

**TTL**: 참조 데이터 성격이 강한 `skills`/`job-categories`/`certs`는 24시간, posting과 더 밀접한 `company-by-skill`은 6시간으로 짧게 잡았다.
