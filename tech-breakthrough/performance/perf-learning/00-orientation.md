# 성능 테스트를 시작하기 전에: 무엇을, 왜 테스트하는가

## 개요

이 문서는 성능 테스트 개념서 트랙의 첫 문서다. 뒤에 이어지는 문서들은 부하 테스트의 개념(가상 사용자, 백분위수, 처리량과 지연의 관계 등)을 설명하는데, 그 설명은 이 프로젝트의 실제 서비스를 예로 든다. 이 문서는 그 바닥을 먼저 깐다. 무엇을 테스트 대상으로 삼는지, 왜 지금 이 시점에 성능 테스트가 필요한지, 그리고 이미 갖춰진 관측 도구가 무엇인지 정리한다.

## 테스트 대상

테스트 대상은 채용 공고 서비스의 백엔드 API다. FastAPI로 만들어졌고, 공고 조회와 기술 스택 통계, 이력서 분석, RAG 챗봇 같은 기능을 REST API로 제공한다. 이 API는 GCP의 작은 가상머신 위에서 돌고, 데이터는 별도의 관리형 Postgres 인스턴스에 저장된다. API 앞단에는 리버스 프록시가 있어 HTTPS를 처리하고, 그 뒤에서 API 서버가 요청을 받아 데이터베이스를 조회하거나 외부 LLM API를 호출해 응답을 만든다.

성능 테스트는 이 경로 전체에 걸쳐 있다. 요청이 프록시를 지나 API 서버에 닿고, API 서버가 데이터베이스에 쿼리를 던지고, 그 결과를 다시 응답으로 돌려주는 전체 과정의 속도와 안정성을 재는 것이 목표다.

## 지금 성능 테스트가 필요한 이유

이유는 세 가지다.

첫째, 지금까지는 기능이 되는지만 확인했다. 공고가 잘 조회되는지, 통계가 맞게 나오는지는 확인했지만, 동시에 여러 사용자가 몰렸을 때도 그렇게 동작하는지는 아직 확인하지 않았다. 데모나 실제 사용에서 여러 명이 동시에 접속하면 평소와 다른 문제가 드러날 수 있다.

둘째, 인프라가 작고 비용에 민감하다. 이 서비스가 도는 가상머신과 데이터베이스는 작은 사양으로 운영되고 있고, 최근에는 비용을 줄이기 위해 데이터베이스 사양을 한 단계 낮추기도 했다. 사양이 작을수록 부하가 걸렸을 때 어디서 먼저 무너지는지 미리 알아두는 것이 중요하다.

셋째, 이미 관측 도구가 갖춰져 있다. 성능을 재려면 보통 새로운 도구를 붙여야 하는데, 이 프로젝트는 API 서버가 이미 모든 요청의 처리량과 지연 시간을 자체적으로 기록하고 있고, 그 값을 실시간으로 그려주는 대시보드도 만들어져 있다. 그래서 부하 테스트 도구만 붙이면 곧바로 결과를 눈으로 볼 수 있는 상태다.

## 이미 갖춰진 관측 도구

## 지켜야 할 원칙

테스트 대상 서버는 데모와 실제 사용을 앞둔 서비스이자 작은 사양으로 운영되는 인프라라는 점을 항상 기억한다. 원칙은 두 가지다.

첫째, 비용이 드는 경로는 건드리지 않는다. 이 서비스의 챗봇 기능은 외부 LLM API를 호출하는데, 반복 요청은 그대로 비용으로 이어진다. 부하 테스트는 비용이 들지 않는 조회 경로만 대상으로 삼는다.

둘째, 프로덕션에는 항상 작게, 짧게 건다. 로컬 개발 환경에서는 마음껏 세게 테스트해도 되지만, 실제 서비스에는 소규모로 짧게만 걸어 다른 사용자나 데모에 지장이 없도록 한다.
