# 부하 테스트가 왜 필요한가: 우리가 직접 겪은 사례로

## 개요

00 문서에서 성능 테스트가 왜 필요한지 미리 짚었다. 이 문서는 그 주장을 실제로 있었던 일로 뒷받침한다. 가상 사용자 3명, 15초짜리 가장 가벼운 부하 테스트가 실제 운영 서비스의 구조적 버그를 잡아낸 과정을, 왜 그런 종류의 버그가 기능 테스트로는 잡히지 않는지와 함께 정리한다.

## 기능은 되어도 규모에서 무너지는 문제

새 기능을 만들고 나면 보통 눈으로 확인한다. 공고 몇 건을 조회해보고 잘 나오면 넘어간다. 문제는 이 확인 방식이 데이터가 적을 때는 통하지만, 데이터가 커지면 전혀 다른 이야기가 된다는 점이다. 부하 테스트 실무에서는 이런 실패를 흔한 유형으로 꼽는다. 시스템은 데이터 규모나 동시 접속이 어느 문턱을 넘을 때만 무너지는데, 정적인 테스트 케이스만으로는 그 문턱 너머의 동작을 검증할 방법이 없다[1]. 몇 달간의 기능 테스트를 통과한 시스템이 실제 트래픽을 받은 지 십수 분 만에 멈춘 사례도 있는데, 원인은 기능이 아니라 데이터베이스 연결 풀처럼 규모에서만 드러나는 지점이었다[1].

우리가 겪은 버그도 같은 성격이다. 공고 목록 조회 기능은 개발 환경의 적은 데이터로는 멀쩡했다. 필터에 걸리는 공고를 전부 가져와서 파이썬에서 페이지를 자르는 구조였는데, 데이터가 적을 때는 그 전부가 몇십 건이라 아무 문제가 없었다. 실제 운영 데이터가 56만 건으로 커진 뒤에야, 그 전부를 가져오는 구조가 데이터베이스의 파라미터 한도를 넘기며 무너졌다. 이런 결함을 부르는 이름이 따로 있다. 필요한 양보다 훨씬 많은 데이터를 가져온 뒤 애플리케이션 쪽에서 자르는 패턴은 도입하기는 쉽지만 알아채기는 어려운 성능 안티패턴으로 분류되고, 작은 데이터의 개발 환경에서는 숨어 있다가 규모가 큰 운영 환경에서만 비용을 드러낸다[2]. 우리 사례는 이 설명과 정확히 일치했다.

## 이런 결함이 방치되기 쉬운 이유

기능 테스트, 코드 리뷰, 심지어 단위 테스트도 이런 결함을 잡지 못한다. 이유는 세 가지 검증이 공통적으로 적은 데이터를 가정하기 때문이다. 단위 테스트는 데이터베이스 대신 가벼운 인메모리 데이터베이스를 쓰고, 테스트 데이터도 사람이 손으로 몇 건만 만들어 넣는다. 코드 리뷰는 로직이 올바른지를 보지, 그 로직이 56만 건을 만났을 때 몇 초가 걸릴지는 계산하지 않는다. 눈으로 하는 수동 확인도 마찬가지로 적은 데이터에서 진행된다.

부하 테스트만이 실제 규모의 데이터와 동시성을 동시에 걸어본다. 우리 사례에서 가상 사용자 3명이라는 아주 작은 동시성조차, 필터 없는 호출 하나가 매번 결정적으로 실패한다는 사실을 곧바로 드러냈다. 동시성이 크지 않아도, 실제 데이터 규모를 상대로 실제 코드 경로를 실행해보는 것만으로 충분했다.

## k6를 선택한 이유

이 결함을 발견하는 데 특별히 복잡한 도구가 필요했던 것은 아니다. 중요했던 것은 스크립트로 반복 가능한 트래픽을 만들 수 있었다는 점이다. k6 스크립트는 파일로 남기 때문에, 같은 시나리오를 로컬에서도 프로덕션에서도, 지금도 나중에도 똑같이 재현할 수 있다. 사람이 매번 브라우저를 열어 클릭하는 방식으로는 이런 재현성을 얻을 수 없다.

임계값 기능도 결정적이었다. `http_req_failed`에 오류율 1퍼센트 미만이라는 조건을 코드에 미리 적어두었기 때문에, 테스트가 끝나자마자 k6가 스스로 실패라고 판정했다. 만약 임계값 없이 터미널에 찍힌 숫자를 사람이 눈으로 훑어봤다면, 11퍼센트라는 오류율을 대수롭지 않게 넘겼을 수도 있다. 통과와 실패를 코드로 미리 정의해 두는 것이, 사람의 판단에 결과 해석을 맡기지 않게 해준다.

마지막으로 이미 갖춰진 관측 스택과의 조합이 컸다. k6가 만든 트래픽을 API 서버가 이미 계측하고 있었고, 그 계측을 데이터베이스 로그까지 따라가 정확한 예외와 코드 경로를 찾아낼 수 있었다. 부하 테스트 도구 하나만으로는 무엇이 실패했는지까지는 알 수 있어도 왜 실패했는지는 알 수 없었는데, 로그와 코드를 함께 봤기 때문에 근본 원인까지 확인하고 고칠 수 있었다.

## 정리

이번 사례가 남긴 교훈은 세 가지다. 기능이 되는 것과 규모에서 버티는 것은 다른 문제이고, 이 차이는 실제 규모의 데이터를 걸어봐야만 드러난다. 부하 테스트 도구는 복잡할 필요가 없지만, 반복 가능하고 임계값으로 통과와 실패를 스스로 판정할 수 있어야 한다. 그리고 가장 작은 규모로 시작하는 원칙이 오히려 더 많은 것을 드러냈다. 가상 사용자 3명이라는 소박한 부하가, 크게 걸었다면 놓쳤을 수도 있는 결정적 실패를 정확히 잡아냈다.
