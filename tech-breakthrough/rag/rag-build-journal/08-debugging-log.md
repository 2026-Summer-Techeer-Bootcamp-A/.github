# 삽질 로그 총정리

이 문서는 데이터 수집부터 벡터화, 프로덕션 적재, RAG 구현, 배포까지 가면서 막혔던 지점과 그것을 어떻게 풀었는지를 한 곳에 모은다. 각 항목은 증상과 원인과 해결과 교훈으로 정리한다. 매칭 관련 삽질은 04 문서에, 적재는 05, RAG는 06, 벡터화는 07에 더 자세히 있고, 여기서는 전체를 한눈에 보도록 요약하며 특히 인프라와 배포에서 겪은 것들을 처음으로 정리한다.

## 1. 데이터와 매칭

한글 매칭이 무너진 것이 가장 큰 사건이었다. 기존 코드는 단어 뒤에 부정 선행 단언을 붙여 경계를 잡았는데, 파이썬의 단어 문자 판정이 한글까지 포함하는 탓에 자바개발자나 파이썬으로 같은 교착어 표현이 전부 매칭에서 빠졌다. 에러가 나지 않아서 한동안 몰랐고, 검출률이 이상하게 낮은 것을 보고서야 드러났다. 한글일 때는 왼쪽만 가드하고 오른쪽 조사를 허용하도록 고쳤다. 여기서 얻은 교훈은 에러가 없다고 정상이 아니라는 것이다. 자세한 내용은 04 문서에 있다.

이 밖에도 한 글자 한글 별칭 뷰가 뷰티 공고에 붙은 오탐, 사이트 푸터의 개인정보 문구를 개념으로 잘못 잡아 93.7퍼센트가 나온 오탐, 옛 페이지 본문에 사이트 네비게이션 메뉴가 통째로 섞여 HTML과 정보보안이 상위를 점령한 오염이 있었다. 셋 다 뿌리는 같았다. 짧거나 모호한 토큰과 본문에 섞인 사이트 구조물이 문제였고, 특정 구만 남기고 본문을 잘라내는 방향으로 풀었다.

## 2. 마트와 벡터화

한 국내 소스는 회사명을 뽑는 방식에서 막혔다. 처음에는 h1 태그를 회사명으로 썼는데, 옛 레이아웃의 h1이 회사명과 공고 제목을 붙여 놓은 형태라 회사명의 93퍼센트가 오염됐다. title 태그를 기준으로 통합 파싱하도록 바꿨고, 시대마다 세 가지로 다른 title 포맷을 모두 처리했다.

마트를 만들 때는 속도가 문제였다. 레코드마다 약 500개의 정규식을 검색하다 보니 39만 건에 5.8시간이 걸렸다. 단일 스레드였기 때문인데, 16코어 중 14개를 쓰는 멀티프로세싱으로 바꿔 약 14배 빨라졌다. 정규식 검색은 CPU를 쓰는 작업이라 GPU로는 못 올리고 프로세스 병렬이 정답이었다.

벡터화에서는 두 가지를 겪었다. 하나는 차원 버그로, 벡터 컬럼이 1536으로 잡혀 있었는데 이는 OpenAI ada-002의 차원이고 우리가 쓰는 BGE-M3는 1024라 적재 자체가 실패할 상황이었다. 임베딩을 만들기 전에 먼저 잡았다. 다른 하나는 입력 설계로, 본문을 통째로 임베딩하면 사이트 크롬 노이즈가 벡터에 들어오고 일곱 소스의 형식 차이가 벡터를 한쪽으로 쏠리게 만들었다. 그래서 본문 대신 정제한 필드를 조합해 입력을 만들었다. 자세한 내용은 07 문서에 있다.

## 3. 프로덕션 적재

적재는 함정이 연달아 나왔다. 첫째로 source_uid 컬럼이 64자였는데 일부 해외 소스와 국내 소스가 URL을 uid로 저장해서, 한 국내 소스는 트래킹 파라미터까지 붙어 691자까지 갔다. 숫자 id만 있던 소량 테스트는 통과했다가 전체를 넣자 터졌고, 컬럼을 Text로 넓혀 풀었다.

둘째로 스키마 교체 덤프에서 막혔다. pg_dump의 clean 옵션이 DROP TABLE보다 먼저 DROP CONSTRAINT를 개별 실행하는데, skill의 기본키를 interest_signal과 resume_skill이 외부 FK로 참조하고 있어서 그 지점에서 실패했다. clean을 버리고 덤프 앞에 명시적으로 DROP TABLE CASCADE를 붙여 풀었다.

셋째로 첫 import가 HTTP 412로 실패했다. gcloud sql import는 Cloud SQL 인스턴스의 서비스 계정으로 GCS를 읽는데 그 계정에 버킷 읽기 권한이 없었다. objectViewer를 부여해 풀었다. 이 세 가지의 전말은 05 문서에 있다.

## 4. RAG와 LLM

RAG에서는 두 가지가 있었다. 하나는 SQL 파라미터 문제로, pool 필터가 NULL일 때 Postgres가 파라미터의 타입을 정하지 못해 AmbiguousParameter 에러가 났다. NULL이 어느 타입인지 모호했기 때문인데, CAST로 타입을 명시해 풀었다.

다른 하나는 LLM 서술이 계속 폴백을 타는 문제였다. 원인이 두 겹이었다. 코드가 비표준 엔드포인트를 쓰고 있었고, 동시에 API 키가 길이 3짜리 placeholder였다. 표준 generateContent 엔드포인트로 고치고 실제 키를 넣자 서술이 살아났다. 흥미로운 점은 이 실패가 오히려 정직성 설계를 증명했다는 것이다. LLM이 죽어도 숫자는 도구가 확정하므로 답이 틀리지 않았고, 서술만 템플릿으로 대체됐다. 자세한 내용은 06 문서에 있다.

## 5. 인프라와 배포

이 구간의 삽질은 다른 문서에 없어서 여기 처음 정리한다. 대부분 클라우드와 컨테이너를 원격에서 다룰 때의 함정이었다.

### env 파일 변경 미반영 문제

Gemini 키를 프로덕션 .env에 넣고 docker compose restart를 했는데 여전히 옛 키를 썼다. restart는 컨테이너를 껐다 켜기만 할 뿐 env_file을 다시 읽지 않기 때문이었다. env를 다시 읽으려면 컨테이너를 재생성하는 up -d를 써야 한다. restart와 up -d의 차이를 정확히 알아야 하는 지점이었다.

### 자동 실행 시 터미널 부재

VM에 인터랙티브로 SSH해서 명령을 붙여넣으려 했는데 셸이 열리지 않았다. 자동 실행 환경은 의사 터미널을 할당하지 않아서, SSH가 접속만 하고 바로 빠져나왔다. 그 결과 붙여넣은 heredoc 블록이 원격이 아니라 로컬에서 실행돼, 로컬에 없는 경로를 찾다가 실패했다. 해결은 대화형 접속을 포기하고 gcloud ssh의 command 인자로 한 줄에 다 넣어 비대화식으로 돌리는 것이었다.

### 따옴표와 백틱 중첩 시 깨짐

Traefik 라우터 설정에는 백틱이 들어가는데, 이것을 gcloud ssh command의 큰따옴표 안에 다시 heredoc으로 넣으니 따옴표와 백틱이 층층이 겹쳐 깨지기 쉬웠다. 파일 내용을 base64로 인코딩해서 전송하고 원격에서 디코딩하는 방식으로 풀었다. base64 문자열은 특수문자가 없어서 어떤 따옴표 안에서도 안전하게 지나간다.

### 브랜치 전환 시 파일 소실처럼 보이는 현상

작업 도중 백엔드의 워킹트리가 다른 기능 브랜치로 여러 번 바뀌었다. 그때마다 방금 만든 RAG 파일이 워킹트리에서 사라진 것처럼 보였는데, 실제로는 커밋이 origin의 main에 안전하게 있고 워킹트리만 다른 브랜치를 보고 있었을 뿐이었다. 원격에 푸시해 둔 커밋은 브랜치 전환으로 사라지지 않는다는 것을 기억하면 당황하지 않는다.

### 머지 충돌의 추가적 특성

기능 브랜치를 main에 병합할 때 config와 main 파일에서 충돌이 났다. 살펴보니 양쪽이 각각 다른 라우터와 설정을 더한 추가성 충돌이라, 한쪽을 버리는 게 아니라 둘을 합치는 것이 정답이었다. CORS 설정만 네이밍이 갈렸는데 더 완전한 쪽을 택했다. 충돌을 보면 먼저 그것이 진짜 대립인지 단순 추가인지 구분하는 것이 중요하다.

### 파이프와 백그라운드 병용 시 출력 소실

백그라운드로 돌린 명령의 출력을 파이프로 넘기면 표준 출력이 유실되는 경우가 반복됐다. 결과를 눈으로 봐야 하는 작업은 파이프 대신 파일로 기록하고 그 파일을 읽는 방식으로 바꿨다.

## 6. 관통하는 교훈

가장 크게 배운 것은 에러가 없다고 정상이 아니라는 점이다. 한글 매칭 붕괴도 차원 버그도 크롬 오염도 모두 예외를 던지지 않고 조용히 잘못된 결과를 냈다. 그래서 수천 건의 실제 결과를 눈으로 보고 분포가 상식과 맞는지 확인하는 습관이 파서 버그를 잡는 유일한 방법이었다.

두 번째는 되돌리기 어려운 작업일수록 비파괴로 설계해야 한다는 것이다. 프로덕션 적재를 일회용 DB에서 먼저 검증하고 타깃 테이블만 건드린 것, 무거운 임베딩 모델을 기능 플래그 뒤에 두어 배포해도 안 켜지게 한 것이 모두 같은 원칙이었다.

세 번째는 원격과 로컬의 경계를 늘 의식해야 한다는 것이다. 배포 구간의 삽질은 거의 전부 명령이 어디서 실행되는지, 어떤 환경 변수를 언제 읽는지를 착각한 데서 나왔다.
