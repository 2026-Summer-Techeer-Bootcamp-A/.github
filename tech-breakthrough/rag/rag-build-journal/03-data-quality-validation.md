# 데이터 품질 검증과 오류 정리

이 문서는 마트 데이터(546k)를 전면 분석해 오류를 찾아내고, 웹에서 확인한 객관적 통계와 대조한 뒤 고쳐나간 과정을 기록한다. 작성일은 2026-07-12다.

## 작업 요약 (한 줄)

마트 데이터를 분석해 오류를 찾아내고, 한국 IT 채용시장의 실제 통계와 비교했다. 그 결과 명백한 쓰레기 레코드는 이번에 바로 지웠지만, 매칭사전 자체의 문제는 고치지 않고 기록만 남겨두었다.

## 대조한 객관적 사실 (웹 검색)

2024년 기준 한국 개발자 채용시장을 보면, 수요가 가장 많은 언어는 JavaScript, Python, Java, TypeScript 순이다. 이 중 Java는 커머스, 결제, 물류, 통신 같은 엔터프라이즈 영역에서 연중 꾸준히 강세를 보이고, 프론트엔드 쪽은 React와 Next.js, 그리고 모던 TypeScript가 주류를 이룬다. Python은 TIOBE, PYPL, IEEE 순위에서 모두 1위권을 지키고 있으며 AI, ML, DevOps 분야에서는 거의 필수 언어로 자리잡았다. 여기에 AI, ML, 데이터, 클라우드, 보안 수요가 급증하는 추세여서 2023년 4분기 AI 관련 공고는 전년 대비 22% 늘었다. 이 내용은 누리조아의 2024 웹개발 취업동향, 프로그래머스의 2023 개발자 설문, 채용 플랫폼의 자체 리포트, JetBrains의 DevEcosystem 2023, dev-korea.com 2025, Nucamp South Korea를 출처로 종합한 것이다.

## 우리 데이터와 대조, 진짜 신호와 오류

국내 종합 채용 소스 공고 전체를 기준으로 시대별 기술 등장률을 살펴보면 아래 표와 같다.

| 기술 | old 2015-2019 | recent 2022-2026 | 판정 |
|---|---|---|---|
| HTML | 46.8% | 3.2% | 오류(노이즈). 구시대 본문에 HTML 잔재 대량 |
| Java | 6.0% | 3.8% | 진짜. Java 강세 = 실제와 일치 |
| Python | 0.7% | 2.3% | 진짜. 상승 추세 = 실제와 일치 |
| JavaScript | 1.1% | 2.2% | 진짜. 상승 = 실제와 일치 |
| React | 0.0% | 0.1% | 오류(추출 누락). 실제론 프론트 최상위인데 0 |
| Spring | 0.4% | 0.8% | 과소. 한국 백엔드 사실상 표준인데 너무 낮음 |
| TypeScript | 0.1% | 0.2% | 과소(React와 같은 원인) |

이 결과를 종합하면, Java의 강세나 Python과 JavaScript의 상승세는 실제 시장 흐름과 정확히 맞아떨어져서 데이터가 살아있다는 신호로 볼 수 있다. 반면 HTML 노이즈, 그리고 React와 Spring과 TypeScript의 과소 집계는 실제 시장과 명백히 어긋나므로 추출 과정의 오류로 판단된다.

## 찾은 오류

### 1. HTML 노이즈 (33% 전체, 구시대 47%)

"HTML"이라는 표기는 이메일 지원 안내문이나 일반적인 웹 언급처럼 기술과 무관한 문맥에도 광범위하게 걸려 들어온다. 이는 taxonomy가 이미 경고한 초일반명 문제이고, CSS와 SQL과 Linux도 같은 성격의 문제를 안고 있다.

### 2. React/Go/Rust/Swift 추출 0 (심각)

이 기술들은 taxonomy의 `_ambiguous_llm_fallback`에 속해 있어서 사전 단독 매칭이 금지되어 있고, 그 결과 텍스트 매칭 모드에서는 아예 잡히지 않는다. 그래서 실제 채용시장에서는 수요가 가장 많은 기술인 React조차 우리 데이터에서는 0에 가깝게 나타나는데, 이는 트렌드 분석과 기술 매칭 모두에 치명적인 문제다.

### 3. Spring/TypeScript 과소

한국 백엔드의 표준이라 할 수 있는 Spring인데도 등장률은 Java의 1/8 수준에 그친다. 스프링, Spring Boot, 전자정부프레임워크 같은 별칭 매칭을 보강할 필요가 있어 보인다.

### 4. close_date 49%가 게시일보다 앞섬

마감일이 게시일보다 앞서는 레코드가 18.5만 건에 달한다. 원인을 따져보면, post_date가 실제 게시일이 아니라 데이터를 수집한 시점을 반영한 값이기 때문에 공고가 만료된 뒤에도 남아 있는 기록이 다수 있는 것이다. 여기에 마감일을 추출하는 정규식이 가끔 엉뚱한 날짜를 잡아내는 문제도 겹쳐 있다.

### 5. 쓰레기 레코드 1,052건

제목이 빈값이거나 구두점만("-", ".", ",")으로 채워진 레코드, 그리고 사이트 고정 문구로 채워진 스텁 보일러플레이트 가짜 공고가 파서를 통과해 들어와 있었다. 특정 파견회사의 특이한 포맷에서는 제목 파싱 자체가 깨지는 문제도 발견되었다.

### 6. 비개발 직군 혼입

AutoCAD, SAP, Photoshop, Illustrator, Orcad, FPGA 같은 기술이 상위권에 올라 있는데, 이는 이 소스가 개발 전문이 아니라 종합 채용 사이트라서 디자인과 하드웨어와 ERP 관련 공고가 원래 많이 섞여 있기 때문이다. 데이터 자체는 진짜지만, 개발 기술 통계를 뽑을 때는 기술직으로 한정하는 필터가 필요하다.

## 고친 것 (이번에 SQL로 즉시 처리, 재빌드 불필요)

이번에는 SQL로 즉시 처리할 수 있는 항목만 정리했고, 재빌드는 필요하지 않았다. 먼저 쓰레기 레코드 1,052건을 fact, raw, tech, cert, category 테이블 전부에서 삭제했다. 그리고 마감일이 게시일보다 앞서는 잘못된 close_date 184,585건을 NULL로 처리했다. 그 결과 해당 소스의 레코드 수는 380,000건에서 378,948건으로 줄었고, 전체 레코드는 545,096건이 되었다.

## 남긴 것 (사전 수정 + 재빌드 필요, 유저 담당 매칭사전 개선 영역)

사전을 고쳐야 하고 그에 따라 재빌드까지 필요한 항목은 이번에 손대지 않고 기록만 남겼는데, 이는 유저가 담당할 매칭사전 개선 영역이다. 첫째로 HTML과 CSS와 SQL과 Linux는 초일반명이라 단독 매칭을 금지하도록 ambiguous 또는 noise 항목으로 강등해야 한다. 둘째로 React, Go, Rust, Swift, TypeScript는 현재 텍스트 매칭에서 완전히 배제되어 있어 등장률이 0으로 나오는데, 문맥가드와 함께 텍스트에서도 매칭되도록 바꾸고 리액트와 스프링 같은 한글 별칭도 보강해야 한다. 이 두 가지를 고치려면 `data-collector-script/taxonomy_v2.json`과 `extract.py`를 수정한 뒤 마트를 재빌드해야 하는데, 재빌드에는 약 50분이 걸린다. 다만 이 매처는 대시보드와 match.py에서도 공유해서 쓰기 때문에, 수정 전에 그 영향도까지 함께 검토해야 한다.

## 기록해둘 한계 (고칠 수 없는 것)

국내 종합 채용 소스의 post_date는 실제 게시일이 아니라 데이터를 수집한 시점을 반영한 값이므로, 연도별 트렌드는 "그 해에 수집된 공고"를 기준으로 잡은 대략적인 값일 수밖에 없다. 그래서 트렌드를 인용할 때는 표본수를 항상 함께 표기해야 한다는 정직성 원칙을 지켜야 한다. 또한 2015년부터 2019년까지의 데이터가 전체의 70%를 차지해서 집계가 구시대 쪽으로 쏠려 있으므로, 최근 흐름을 제대로 보려면 연도별로 나누어 살펴야 한다.
