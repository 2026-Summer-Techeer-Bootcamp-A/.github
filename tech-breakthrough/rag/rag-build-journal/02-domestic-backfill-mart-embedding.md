# 데이터 정규화와 임베딩 준비

이 문서는 국내 채용 공고 데이터를 확보해 마트에 넣고, 프로덕션 적재와 임베딩을 준비한 과정을 적는다. 날짜는 2026-07-11에서 07-12까지다.

## 작업 요약 (한 줄)

국내 공고 데이터 39만 건을 추가로 확보해 마트에 넣는 작업을 진행하고 있으며, 임베딩 차원 버그를 고쳤고, 프로덕션 적재 방법을 정했다.

## 1. 마트 통합

국내 채용 공고 raw 데이터는 그대로 쓸 수 없어서, 조회용 SQLite 마트로 정규화해야 점수화와 검색이 가능해진다. 이를 위해 정규화 파이프라인에 국내 데이터 처리 로직을 추가했고, 경력을 나타내는 한글 문자열(예: "경력3년이상")은 min/max 숫자로 바꾸고, 게시 시점을 나타내는 값을 정리했다. 기술 스택은 본문 텍스트에서 taxonomy 매칭으로 뽑는다.

처음에는 단일 스레드로 돌렸는데, 레코드마다 약 500개 정규식을 검색하다 보니 39만 건을 처리하는 데 5.8시간이 걸렸다. 그래서 롤백한 뒤 멀티프로세싱(16코어 중 14개 사용)으로 다시 돌렸고, 약 14배가 빨라졌다.

현재 진행 중인 결과는 기존 마트 166,148건에 새로 확보한 국내 공고 391,351건이 더해져 약 55만 건에 이른다. 배치마다 커밋하도록 만들어서 중간에 멈춰도 이미 처리한 분량은 남는다.

## 2. 임베딩 차원 버그 수정

벡터를 저장할 컬럼이 `Vector(1536)`으로 잡혀 있었는데, 1536은 OpenAI ada-002의 차원이고 우리가 쓸 모델은 BGE-M3라 1024차원이 맞다. 이 숫자가 맞지 않으면 적재 자체가 실패한다.

그래서 `backend/app/core/config.py`에 `embedding_dim=1024`를 두고 모델과 더미 생성기가 이 값을 참조하도록 고쳤고, 하드코딩된 1536은 전부 없앴다. alembic 마이그레이션이 따로 없어서 모델 정의가 곧 테이블 정의이므로, 모델만 고치면 충분했다.

이 버그를 지금 고쳐야 했던 이유는 이게 임베딩 생성과 프로덕션 적재로 넘어가는 관문이기 때문이다. 고치지 않은 채로 임베딩부터 만들었다면, 몇 시간 들여 만든 벡터를 적재 단계에서 다 넣지 못하는 상황이 생겼을 것이다.

## 3. 프로덕션 적재 계획 (아직 실행 전)

원칙은 무거운 계산은 로컬에서 하고 프로덕션에는 결과물만 싣는다는 것이다. 수집, 추출, 임베딩은 전부 CPU나 GPU를 쓰는 작업이라 로컬에서 처리하고, 프로덕션은 완성된 데이터만 받는다.

저장소는 Cloud SQL for Postgres에 pgvector를 쓴다. 이미 백엔드 RAG가 쓰는 설계이므로 새로운 기술을 들이는 것은 아니다.

적재 방법은 마트와 임베딩을 CSV로 뽑아 GCS에 올린 다음, Cloud SQL로 COPY 적재하는 방식이다. 55만 행 자체를 넣는 데는 몇 분이면 충분하다.

주기는 데모용이라 일회성 시드로 충분하고, 발표 직전에 한 번 더 갱신하면 된다.

중요하게 구분해야 할 지점은, RAG 검색과 chat 로직(⑤)이 "옮기는 데이터"가 아니라는 것이다. 이는 프로덕션 서버가 적재된 DB를 향해 요청 시점에 도는 앱 코드이기 때문에, 프로덕션 이전 단계의 관문은 마트와 임베딩이라는 데이터뿐이고 RAG 앱은 데이터를 옮긴 뒤에 배포하면 된다.

## 4. 임베딩 계획

모델은 BGE-M3를 쓰고 dense 1024차원만 사용한다(sparse와 colbert는 쓰지 않는데, pgvector에는 dense만 필요하기 때문이다). 범위는 유저 결정에 따라 전체 55만 건으로 잡았고, 옛 공고까지 유사검색 대상에 넣는다. 장비는 로컬 GPU RTX 4060을 쓴다.

RTX 4060에서 dense-only, FP16 기준으로 예상 시간을 가늠해보면, 본문을 512토큰으로 자르고 배치를 키우면 대략 초당 100~200건이 나와 55만 건에 약 1시간이 걸린다. 반대로 본문을 길게(1024토큰+) 넣거나 배치가 작으면 약 2~3시간이 걸린다. 결론적으로 대략 1~2.5시간 사이가 될 것이고, 자르는 길이가 가장 큰 변수다.

속도를 높이려면 본문을 512토큰으로 자르는 게 좋은데, 공고 매칭 신호는 제목과 본문 앞부분에 있기 때문이다. 여기에 FP16을 쓰고, VRAM 8GB 한도에서 배치를 최대로 잡고, dense-only로 설정하면 된다.

## 진행 현황 (퍼센트)

- ① 수집: 95% (국내 공고 확보 1차 100%, 2차 잔여)
- ② 마트화: 진행 중 (국내 공고 적재 중, 곧 완료되면 약 55만 건)
- ③ 프로덕션 적재: 5% (방법만 결정)
- ④ 임베딩: 10% (생성 0%, 차원 버그 수정 완료)
- ⑤ RAG 파이프라인: 0%

크리티컬 패스는 마트 완료 다음 임베딩 생성(GPU를 쓰는 가장 오래 걸리는 구간) 다음 Cloud SQL 적재 다음 RAG 배포 순서다.

## 관련 파일

- 차원: `backend/app/core/config.py`, `backend/app/models/posting.py`
