# 실데이터 적재: mart에서 Postgres로 (ETL)

## 개요

이 문서는 구현기(build journal) 트랙의 두 번째 문서로, RAG 구현의 첫 실작업인 데이터 적재를 기록한다. 지금까지 개발 DB는 더미 데이터로 채워져 있었지만, RAG의 임베딩, 벡터 검색, 그래프, 에이전트는 모두 실제 채용 공고를 대상으로 동작해야 의미가 있으므로, 바닥이 되는 실데이터부터 먼저 정확히 적재해야 했다. 그래서 이 문서는 무엇을 왜 그렇게 옮겼는지와 적재 후 검증 결과를 실측으로 남긴다.

## ETL의 정의

ETL은 Extract(추출), Transform(변환), Load(적재)의 약어로, 한 저장소의 데이터를 다른 저장소가 요구하는 형태로 옮기는 데이터 파이프라인의 기본 골격을 이룬다. 이번 작업에서 세 단계가 각각 무엇을 뜻하는지는 다음과 같다.

- Extract(추출): 원본 저장소에서 데이터를 읽는 단계로, 이 작업의 원본은 데이터 수집과 정규화를 처리하는 파이프라인이 산출한 `mart.db`(SQLite)이며, 이미 dedup과 표준화가 끝난 정본이다.
- Transform(변환): 원본의 스키마를 목적지 스키마에 맞게 바꾸는 단계인데, mart가 이미 깨끗한 상태이므로 이 작업의 변환은 재정규화가 아니라 형태를 맞추는 일에 가깝다. 주요 변환은 아래 "무엇을 어떻게 변환하는가" 절에 정리한다.
- Load(적재): 변환한 데이터를 목적지인 백엔드 도커의 PostgreSQL에 밀어 넣는 단계로, 12만 건 규모이므로 한 건씩이 아니라 배치로 넣는다.

이 작업이 순수 적재에 가까운 이유는 정규화와 중복 제거가 상류의 데이터 수집 단계에서 이미 끝났기 때문이다. 따라서 이 단계의 진짜 산출물은 옮긴 데이터가 원본과 건수, 분포까지 일치하는지 확인하는 검증이다.

## 현재 진행 단계

전체 RAG 로드맵(T0 기반 → T1 그래프 → T2 벡터 → T3 에이전트 → T4 데모 → T5 평가)에서 이 작업은 T0 기반 단계에 속하며, 임베딩(T2)이 이 적재로 DB에 들어온 공고를 대상으로 벡터를 만드는 작업인 만큼 적재는 임베딩의 선행 조건이 된다. 즉 이 단계는 RAG의 지능 자체가 아니라, 그 지능이 딛고 설 바닥을 까는 작업이다.

## 변환 대상과 방법

원본(mart)과 목적지(Postgres)의 핵심 매핑은 다음과 같으며, 전체 매핑표와 근거는 설계 문서(`docs/superpowers/specs/2026-07-10-mart-to-postgres-etl-design.md`)에 있다.

| 변환 | 규칙 |
|---|---|
| `posting_id` 분해 | `source:id` 형태의 문자열을 `:` 기준으로 나눠 `source`와 `source_uid` 컬럼에 각각 저장한다. |
| `pool` 도출 | 원본에 없는 값을 source로부터 만든다. 국내(domestic) 소스와 해외(global) 소스를 각각 묶어 만든다. |
| tech → skill_id | `fact_posting_tech`의 표준 기술명을 skill 사전의 id로 바꿔 링크 테이블에 넣는다. 사전은 taxonomy와 mart의 distinct 값을 합쳐 시딩한다. |
| region | 단일 텍스트를 `region_city`에 원문 그대로 보존한다. 국내면 `region_country`='KR', 그 외 NULL. 위경도, 구/군 정보는 이번 적재에서 비운다(mart에 없음). |

## 적재 순서

로더는 다음 순서로 동작하는데, 순서가 중요한 이유는 링크 테이블(기술, 자격증, 직군)이 공고와 사전의 id를 참조하기 때문이며, 그래서 공고와 사전이 먼저 존재해야 한다.

1. 파기: 기존 더미 데이터를 TRUNCATE로 비우되, 스키마(테이블 구조)는 유지한다.
2. 사전 시딩: 기술(skill), 자격증(cert), 직군(job_category) 사전을 먼저 채운다.
3. 공고 적재: `posting`과 원문 보관용 `raw_posting`을 넣는다.
4. 링크 적재: 공고와 사전을 잇는 기술, 자격증, 직군 링크 테이블을 넣는다.
