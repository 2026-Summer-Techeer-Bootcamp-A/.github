# 데이터 매칭이 왜 이렇게 힘들었나

이 문서는 공고 텍스트에서 기술과 개념을 뽑아내는 매칭이 얼마나 어려웠는지를 적는다. 날짜는 2026-07-12다.

## 한 줄 요약

매칭은 사전에 단어를 넣고 텍스트에서 찾는 것처럼 단순해 보이지만, 실제로는 그렇지 않았다. 한국어, 모호한 약어, 사이트 노이즈 세 가지가 계속 발목을 잡았다.

## 매칭이 필요한 이유

우리 데이터는 채용공고 원문이라 "우리는 Java, Spring, MySQL 경험자를 찾습니다" 같은 자유 텍스트로 되어 있는데, 이걸 그대로 두면 집계도 검색도 되지 않는다. 그래서 사전을 만들어 원문을 정규 명칭으로 바꾸는 작업을 거친다. Java, 자바, java 를 전부 하나의 Java 로 모으는 식이며, 이 정규화가 점수화와 트렌드 분석으로 가는 관문이 된다.

방식은 룰 기반으로, 사전에 별칭을 넣고 정규식으로 찾는다. LLM을 쓰지 않는 이유는 세 가지인데, 공짜이고 재현되며 과거 데이터에도 소급 적용된다는 점이다. 38만 건을 매번 LLM으로 돌리면 돈과 시간이 감당되지 않기 때문이다.

## 함정 1. 한국어 교착어의 단어경계 붕괴

가장 크고 늦게 발견한 문제다.

영어는 단어를 띄어 쓰기 때문에 "java developer"처럼 java 와 developer 사이에 공백이 있고, 그래서 단어경계 매칭이 잘 먹는다. `java` 양옆에 영숫자가 없을 때만 잡으면 되므로 javascript 안의 java 는 잡히지 않는다. 정확한 방식이다.

한국어는 다르다. 조사와 명사가 딱 붙어서 "자바개발자", "파이썬으로", "리액트기반", "토익점수"처럼 공백이 없는 경우가 흔하다.

기존 코드는 한글에도 이 단어경계를 그대로 써서 `(?<!\w)자바(?!\w)` 이런 식으로 짜여 있었다. 문제는 파이썬 정규식에서 `\w` 가 한글도 포함한다는 점이었다. 그래서 "자바개발자"는 자바 뒤에 개라는 한글이 오는 순간 매칭이 깨져버렸다.

실제로 테스트해보니 이랬다.

```
자바 개발자   O
자바개발자    X   <- 공백 없으면 실패
자바스크립트   X
파이썬으로    X
파이썬개발    X
리액트기반    X
토익점수     X
```

이게 무슨 뜻이냐면, 공백 없이 조사나 명사가 붙은 한국어 공고에서는 기술을 거의 못 뽑아냈다는 뜻이다. 한국 공고는 "자바 개발"보다 "자바개발"이 더 흔한데, 그 결과 리콜이 조용히 무너지고 있었는데도 에러가 나지 않아서 몰랐다.

이것이 03 문서에서 본 React 0퍼센트, Spring 과소의 진짜 배경 중 하나였다. 사전에 별칭이 있어도 매칭 규칙이 한국어를 받아주지 못했던 것이다.

### 해결 방법

그래서 한글 별칭에는 규칙을 따로 주기로 했다. 왼쪽만 가드를 걸어 한글이나 영숫자가 앞에 붙어 있으면 다른 단어의 일부로 보고 매칭에서 제외하고, 오른쪽은 열어서 조사와 접미사가 붙는 것을 허용한다. 이렇게 하면 "파이썬으로", "자바개발자"가 다 잡힌다.

## 함정 2. 자바와 자바스크립트

오른쪽을 열어주니 새 문제가 생겼다. "자바스크립트" 안의 "자바"가 Java 로 잡혀서, JavaScript 공고가 Java 공고로도 세어지는 중복 계상이 발생한 것이다.

영어에는 이 문제가 없었다. "javascript"는 java 뒤에 script라는 영숫자가 붙어서 단어경계가 이를 막아주기 때문이다. 한글만 이 문제가 남았다.

### 해결 방법

접두사 충돌을 자동으로 막는 방식으로 풀었다. 어떤 별칭 A가 다른 별칭 B의 앞부분이면, A 매칭에 "B의 나머지가 뒤에 오면 안 된다"는 조건을 붙이는 것이다. 자바는 자바스크립트의 앞부분이므로, 자바 뒤에 "스크립트"가 오면 자바 매칭을 취소하도록 하면 `자바(?!스크립트)`가 된다.

결과.

```
자바 개발자   -> Java
자바개발자    -> Java
자바스크립트   -> JavaScript   (Java 아님)
파이썬으로    -> Python
리액트기반    -> React
```

이건 자바 하나만 손보는 방식이 아니다. 사전 전체에서 접두사 관계를 자동으로 찾아 규칙을 만들기 때문에, 새 별칭을 추가해도 알아서 충돌을 막아준다.

## 함정 3. 모호한 맨토큰

React, Go, Rust, Swift, C, R. 이 단어들은 사전에 있어도 자유 텍스트에서 단독으로 매칭하면 안 되는데, "go"는 영어 동사이고 "r"은 그냥 알파벳이며 "swift"는 빠르다는 뜻이고 "react"는 반응하다는 뜻이기 때문이다.

그래서 이 토큰들은 모호어로 격리해 텍스트 매칭에서 빼버리고, 구조화 필드에서만 신뢰하도록 했다.

문제는 이거다. 한국 채용시장에서 React 는 프론트 최상위 수요인데, 우리 데이터에서는 0에 가까웠다. 격리 때문에 아예 뽑히지 않았던 것이다.

이건 트레이드오프다. 격리를 풀면 노이즈가 들어오고, 격리를 유지하면 진짜 신호를 놓친다.

### 접근 방법

두 갈래로 나눴다.

첫째, 한글 별칭은 살리기로 했다. "리액트", "러스트"는 모호하지 않고, 한글로 쓰면 프레임워크 이름이 거의 확실하기 때문이다. 그래서 한글 별칭은 정규 사전에 두고 매칭했으며, 함정 1을 고치면서 이 부분은 자동으로 회복됐다.

둘째, 국내 소스에 한해 영문 모호어 일부를 복원했다. 국내 종합 채용 소스는 한국어 IT 공고라 "react", "swift"가 프레임워크일 확률이 높기 때문에, 그 소스의 파서에서만 React, Swift, Rust, Go 를 안전 목록으로 되살렸다. 해외 영어 소스에는 적용하지 않았다.

## 함정 4. 사이트 크롬 노이즈

국내 종합 채용 소스의 본문을 열어보면 이런 게 섞여 있다.

```
로그인 회원가입 기업 서비스 채용정보 검색 마이페이지 공지사항 ...
```

이건 공고 내용이 아니라 웹페이지의 메뉴, 버튼, 푸터다. 옛 페이지의 HTML을 그대로 가져오면 이 껍데기가 본문에 딸려 온다.

여기에 "HTML", "CSS" 같은 단어도 카테고리 라벨로 박혀 있어서, HTML 이 전체의 33퍼센트에서 잡혔고 구시대 공고에서는 47퍼센트에 달했다. 말이 안 되는 수치인데, 진짜 HTML 수요가 아니라 사이트 메뉴를 센 것이었다.

### 해결 방법 1차

본문에서 네비게이션 마커를 찾아 그 뒤를 잘라내는 방식으로 1차 수정을 했다. "관심 키워드" 같은 마커가 나오면 거기서 본문을 끊는 식이다. 최근 공고는 이걸로 깨끗해졌지만, 전체 집계를 보니 HTML이 여전히 1위였다.

### 함정 4-1. 구시대 네비게이션 메뉴 통째 유입

연도별로 쪼개보니 원인이 드러났다. HTML 매칭이 2014년부터 2017년에만 몰려 있었는데, 2017년 한 해에 7만 3천 건이었다가 2018년부터는 백 건대로 뚝 떨어졌다. 개념 정보보안도 똑같은 연도 분포를 보였고, 두 개의 연도별 카운트가 거의 똑같았다.

이건 우연이 아니었다. 2014년부터 2017년의 옛 페이지는 레이아웃이 달라서 공고 본문에 사이트 네비게이션 메뉴 전체가 딸려 왔는데, 그 메뉴에는 웹 개발, 네트워크, 보안 같은 직무 카테고리 목록이 있었다. 그래서 실제 공고 직무와 무관하게 모든 페이지에 HTML과 정보보안이 박혔고, 2017년이 전체의 32퍼센트를 차지했기 때문에 이 크롬이 집계를 통째로 점령한 것이다.

### 함정 4-2. 마커를 리스트 순서로 자른 버그

메뉴 시작 마커를 추가했다. "메뉴 건너뛰기", "채용정보 홈" 같은 것들이다. 그런데 2015년과 2016년은 여전히 잡히지 않았다.

원인은 자르는 로직에 있었다. 마커를 리스트 순서대로 훑다가 처음 발견한 마커에서 잘랐는데, 문제는 텍스트상 위치가 아니라 리스트 순서로 골랐다는 점이었다. 리스트 앞쪽 마커가 텍스트에서는 뒤에 나오면 엉뚱하게 늦게 잘렸고, 그래서 앞에 있던 메뉴가 그대로 남았다.

그래서 모든 마커의 위치를 다 구한 뒤 가장 이른 위치에서 자르도록 고쳤다. 리스트 순서가 아니라 텍스트 위치 기준으로 바꾼 것이다.

결과.

```
        1차(리스트순)   2차(가장 이른 위치)
2015    HTML 68%        HTML 0.1%
2016    HTML 52%        HTML 0.2%
2017    HTML 6%         HTML 0.2%
2023    HTML 0%         HTML 0.0%
```

전 연도가 깨끗해졌다. HTML은 사이트 메뉴가 아니라 진짜 웹 스킬만 남았고, 12년 트렌드를 이제 믿을 수 있게 됐다.

교훈이 하나 더 있다. "처음 찾은 것"과 "가장 이른 것"은 다르며, 마커가 여러 개면 위치로 골라야 한다. 이것도 샘플 한 건으로는 절대 잡을 수 없고, 연도별로 쪼개서 분포를 봐야 드러났다.

## 함정 5. 기술보다 어려운 개념

기술은 그나마 단일 토큰이다. Kafka, Redis, Docker 같은 식이다. 반면 개념은 구인데, "대규모 트래픽", "마이크로서비스 아키텍처", "이벤트 기반" 같은 것들이다.

여기서 최악의 함정은 짧은 맨토큰이다. "AI" 하나만 사전에 넣으면 "AI 시대", "AI가 바꾼다", 회사 소개의 "AI 기업"까지 다 잡혀서, 개념 수요와 무관한 마케팅 문구가 전부 들어온다. "대규모"도 마찬가지로, "대규모 공채", "대규모 투자"가 기술 개념으로 오인된다.

### 접근 방법

사전 설계로 막기로 했다. 복잡한 문맥 엔진을 만드는 대신, 모호한 맨토큰을 별칭에서 아예 금지하는 방식이다.

"AI" 단독은 금지고, "생성형 AI", "AI 모델", "AI 서비스"처럼 문맥이 실린 구만 넣는다. "대규모" 단독도 금지고, "대규모 트래픽", "대규모 서비스", "대용량 트래픽"만 넣는다.

별칭 하나하나가 그 자체로 충분히 구체적이면 노이즈가 원천적으로 들어오지 않는데, 이 방식이 문맥 판정보다 단순하면서도 재현된다.

소프트스킬은 개념축에서 통째로 뺐다. 협업, 소통, 책임감, 열정 같은 것들은 모든 공고에 다 있어서 변별력이 0이고, 넣으면 그래프가 다 똑같아 보이기 때문이다. 그래서 배제했고, 배제한 목록은 사전에 근거로 남겨두었다.

## 함정 6. 뷰와 뷰티

사전을 고치고 국내 종합 채용 소스에 실제로 돌려보니 Vue가 1위로 나왔다. 1.68퍼센트로 Java 나 React 보다 높았는데, 말이 안 되는 결과였다.

무엇이 잡혔는지 뽑아봤다.

```
일본 뷰티 S&OP 운영
글로벌 뷰티 코스메틱 제품
뷰티 미용 지도보기
아테나뷰티 에스테틱
뷰티매니저 매장관리
```

전부 "뷰티"였다. Vue 의 한글 별칭에 1글자 "뷰"가 있어서 "뷰티"의 앞글자 "뷰"가 Vue 로 잡힌 것이었다. 이 소스는 종합 채용 사이트라 뷰티, 미용 공고가 많은데, 그래서 화장품 공고가 프론트엔드 프레임워크로 둔갑했다.

교훈은 명확했다. 1글자 한글 별칭은 위험하므로 "뷰"를 지웠고, Vue 는 vue, vue.js, 뷰js 로만 잡도록 했다. 지운 뒤 Vue 는 상위권에서 사라졌다.

## 함정 7. 개인정보 처리방침

개념 사전도 실제로 돌려봤는데, 개인정보 컴플라이언스 개념이 93.72퍼센트로 나왔다. 공고 열 개 중 아홉 개가 개인정보를 다룬다는 뜻인데, 당연히 거짓이었다.

무엇이 잡혔는지 봤다.

```
공고 개인정보 수정 이직을 준비하는
포지션 제안 스크랩 공고 개인정보 수정
```

"개인정보 수정"이었다. 이건 공고 내용이 아니라 해당 페이지의 메뉴 링크로, 모든 페이지 하단에 개인정보처리방침이 붙기 때문에 맨토큰 "개인정보"가 거의 모든 공고에 걸렸던 것이다. 함정 4의 사이트 크롬이 개념 사전에서 다시 터진 셈이다.

맨토큰 "개인정보"를 지우고 법적, 기술적으로 구체적인 구만 남기는 방식으로 고쳤다. "개인정보보호법", "개인정보 비식별", "gdpr", "isms" 같은 것들이다. 이러면 푸터 메뉴는 걸리지 않고 실제 업무만 잡히는데, 고친 뒤 93.72퍼센트에서 0.46퍼센트로 떨어졌다.

## 검증 결과: 한국어 매칭 수정의 성과

국내 개발 채용 소스로 대조해봤다. 이 소스는 담당업무, 자격요건 같은 실제 텍스트 필드를 구조화해서 주고 사이트 크롬이 없어서, 여기서 한국어 매칭 수정의 효과가 그대로 드러났다.

```
Python 27.6%   C++ 20.5%   Linux 19.7%   Git 16.3%
Java 13.4%     JavaScript 12.8%   React 8.7%   Spring 8.1%
Kubernetes 7.9%   TypeScript 7.1%
```

React 가 8.7퍼센트로 살아났다. 전에는 0에 가까웠던 수치다. Spring, TypeScript 도 정상 범위로 올라왔고, 03 문서에서 확인한 웹 벤치마크와 맞는 분포가 됐다.

개념도 이 소스에서 건강하게 나왔다.

```
확장성 성능 21.2%   생성형 AI LLM 11.9%   머신러닝 딥러닝 11.8%
CI/CD 10.6%   테스트 품질 10.1%   대규모 트래픽 5.9%   MSA 2.8%
```

반면 국내 종합 채용 소스는 기술 검출률이 낮게 나온다. 이건 버그가 아니라, 그 소스가 개발 전용이 아니라 종합 채용 사이트이기 때문이다. 뷰티, 마케팅, 영업, 생산직 공고가 대다수라 개발 기술 밀도 자체가 낮은 게 맞다. 그래서 그 소스는 다년치 트렌드용으로 쓰고, 추천은 국내 개발 채용 소스와 다른 국내 소스를 쓴다.

## 정리

매칭은 사전 문제가 절반이고 규칙 문제가 절반이었다. 사전에 별칭을 아무리 많이 넣어도 한국어 규칙이 틀리면 다 놓쳤고, 그렇다고 규칙을 열면 노이즈가 들어왔다. 결국 언어의 특성을 하나씩 실측으로 확인하면서 규칙을 다듬는 수밖에 없었다.

교훈은 03 문서와 같다. 샘플 한두 건으로는 절대 잡을 수 없고, 수천 건을 실제로 돌려서 분포를 눈으로 봐야 드러난다. 에러가 나지 않기 때문에 오히려 더 위험하다.
